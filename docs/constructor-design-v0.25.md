# Конструктор: проект раздела «собери свою таблицу из данных проекта»

Только проектирование, без применения. Цель — дать инженеру **собрать любой документ
под себя** (таблицу для заказчика, спецификацию, ведомость, опросный лист) из данных,
которые уже живут в программе: теги, оборудование и его параметры, справочники, проект,
закупки. Собрал один раз — сохранил как шаблон, применил к любому проекту, обновил
данные одной кнопкой, выгрузил в Excel/PDF/Word или в Проводник программы.

Главный принцип: **не «ещё одна выгрузка», а живой документ**. Ячейки помнят, откуда
взяты данные, и умеют обновляться при изменении проекта — но пользователь всегда может
править всё руками, как в обычном Excel.

Всё остаётся локальным и офлайн (Electron + Express + SQLite) — никаких CDN и внешних
сервисов.

> **Статус внедрения**: Фаза 1 (MVP) реализована — раздел «Конструктор» в меню,
> Библиотека (Продолжить/Мои/Общие/Корзина), редактор на движке Univer (спайк
> Фазы 0 пройден: снапшот в БД, RU-локаль, ленивый чанк), мастер «Собрать данные»
> (теги/оборудование → каталог полей → фильтр → вставка), автосейв, Общий/Личный,
> именование при закрытии, экспорт XLSX (скачать / в Проводник). Фаза 2 (умные
> блоки, MVP) тоже реализована: вставка мастером создаёт блок с запросом,
> панель «Блоки» с «Обновить/Обновить все/Отвязать», сверка по entityKey
> (+новые/−выпавшие/изменённые), ручные правки сохраняются как overrides
> (детекция сравнением с последней записью, а не перехватом команд — упрощение),
> конфликты «моё против из проекта» поштучно и массово, значок устаревания по
> отпечатку данных. Зеркало-файл в Проводнике (часть III §5) и печать/PDF (MVP)
> реализованы. Совместное редактирование (часть IV, MVP) реализовано: комната
> документа через socket.io, аватарки участников («✏️ N в документе»), цветные
> рамки выделений коллег поверх канваса, живая репликация мутаций движка с
> гашением эха через fromCollab. Упрощения MVP относительно части IV: репликация
> без OT-трансформации (конкурирующие структурные правки редки, автосейв — LWW),
> op-лог не персистится, bindings блоков не синхронизируются между открытыми
> сессиями до переоткрытия. Впереди: OT-очередь с ревизиями, полная пагинация
> печати, свободные строки внутри блока через перехват команд, DOCX.

> Это часть I — обзор и решения. Глубокая проработка подсистем (форматы привязок,
> алгоритм сверки, грамматика путей, пагинация печати, блокировки, лимиты, тест-план,
> реестр крайних случаев) — в **`constructor-design-v0.25-deep.md`** (часть II).
> Пространство раздела, «мои/общие», просмотр/редактирование, именование при закрытии
> и зеркало в Проводнике — в **`constructor-design-v0.25-space.md`** (часть III;
> она уточняет §1 «Библиотека» этой части). Совместное онлайн-редактирование
> несколькими пользователями — в **`constructor-design-v0.25-collab.md`** (часть IV;
> она заменяет блокировку из части II §10 для сетевого режима).

## 1. Место в программе

Новый пункт бокового меню — **«Конструктор»** (иконка `Table2`/`LayoutGrid`). Внутри две
зоны:

1. **Библиотека** — список документов и шаблонов текущего проекта (карточки/список:
   название, автор, дата изменения, метка «шаблон», превью-миниатюра). Кнопки:
   «Новый документ», «Собрать данные» (мастер, §5), «Из шаблона».
2. **Редактор** — полноэкранное «пространство Экселя»: панель инструментов, строка
   формул, сетка ячеек, ярлыки листов внизу, правая панель «Данные проекта» (§6).

Документ Конструктора — это **книга** (workbook): несколько листов, у каждого листа —
сетка, стили, формулы, привязки к данным. Хранится в БД проекта, привязан к `projectId`.

## 2. Ядро таблицы: свой грид или готовый движок

Написать «настоящий Excel» с нуля (формульный движок, пересчёт зависимостей, фильтры,
условное форматирование, undo/redo, буфер обмена совместимый с Excel) — месяцы работы и
вечная погоня за багами. Правильный ход — взять открытый движок и нарастить сверху наш
слой данных.

| Вариант | Лицензия | Формулы | Фильтры/сорт. | Форматирование | Состояние | Вердикт |
|---|---|---|---|---|---|---|
| **Univer** (@univerjs/*) | Apache-2.0 (ядро) | полный движок, сотни функций, **кастомные функции** | есть | шрифты, цвета, границы, объединение, условное форматирование, числовые форматы | активно развивается, React-обвязка | **рекомендую** |
| Fortune-Sheet | MIT | базовый набор | есть | есть | форк Luckysheet, поддержка вялая | запасной |
| Handsontable + HyperFormula | коммерческая / GPLv3 | хороший | есть | есть | платно для нас | нет |
| x-spreadsheet / jspreadsheet CE | MIT | слабые | частично | ограничено | заморожены | нет |
| Свой грид | — | всё писать самим | самим | самим | — | только если Univer не взлетит |

**Рекомендация: Univer.** Что важно для нас:

- ставится как npm-пакеты, работает полностью офлайн — совместимо с portable-сборкой;
- формульный движок с регистрацией **собственных функций** — ровно то, что нужно для
  `=ТЕГ()`, `=ПАРАМ()` и прочих (§7);
- снапшот книги — обычный JSON (`IWorkbookData`): легко хранить в SQLite, диффать,
  версионировать;
- в открытой части есть: сетка, стили, объединение, формулы, фильтры, сортировка,
  условное форматирование, числовые форматы, поиск/замена, undo/redo, заморозка областей.

Честные оговорки (учтены в плане):

- часть «взрослых» модулей у Univer платная (сводные таблицы, их импорт/экспорт, печать).
  Экспорт/печать делаем **своими руками** через серверный рендер (§10) — это даже лучше,
  т.к. полностью контролируем результат;
- бандл заметный (~1.5–2 МБ gzip) → раздел грузим **лениво** (dynamic import route-чанком),
  на старте приложения он не стоит ничего;
- версию пиним точно, обновляем осознанно (снапшот-формат должен оставаться читаемым).

Решение по движку фиксируем после «спайка» на 1–2 дня: собрать прототип с кастомной
функцией и снапшотом в SQLite. Если что-то не так — Fortune-Sheet тем же интерфейсом
(наш слой данных от движка не зависит, см. §4 про адаптер).

## 3. Модель данных (Prisma)

```prisma
model ConstructorDoc {
  id          String   @id @default(uuid())
  projectId   String
  name        String
  kind        String   @default("DOC")      // "DOC" | "TEMPLATE"
  description String   @default("")
  workbook    String                        // JSON-снапшот книги (движок)
  bindings    String   @default("[]")       // JSON: умные блоки и привязки (§6)
  settings    String   @default("{}")       // JSON: параметры печати, ориентация, поля
  createdById String?
  updatedById String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
  versions    ConstructorDocVersion[]

  project     Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  @@index([projectId, kind])
}

model ConstructorDocVersion {
  id        String   @id @default(uuid())
  docId     String
  version   Int
  workbook  String                          // снапшот на момент версии
  bindings  String   @default("[]")
  comment   String   @default("")           // "до обновления данных", "ручное сохранение"
  authorId  String?
  createdAt DateTime @default(now())

  doc       ConstructorDoc @relation(fields: [docId], references: [id], onDelete: Cascade)
  @@index([docId, version])
}
```

Почему **`bindings` отдельно от `workbook`**: снапшот — собственность движка (меняется
при апгрейде), а привязки — наша бизнес-логика. Привязки ссылаются на диапазоны через
именованные диапазоны/якоря листа, а не через «сырые» координаты, чтобы вставка строк
выше блока их не ломала.

Версии: автоматический снимок **перед каждым «Обновить данные»** и перед применением
шаблона + ручное «Сохранить версию». Храним последние N (настройка, по умолчанию 20),
старые схлопываем. Это дешёвая страховка от «обновил и всё поехало».

Автосохранение — как в карточке тега: дебаунс 1.5–2 с после последнего изменения,
плюс сохранение при уходе с экрана. Никаких кнопок «Применить».

## 4. Слой данных проекта: каталог полей

Ключевая проблема: параметры оборудования у нас **динамические**
(`ComponentElement.specs = { groups: [{title, params: [{key, value, unit}]}] }`,
`Tag.metadata` — произвольный JSON). Заранее «списка колонок» не существует — его надо
**строить по факту**.

Новый серверный сервис — **каталог полей** (`GET /api/constructor/catalog?projectId=...`):
сканирует данные проекта и возвращает дерево доступных полей:

```
Тег                    ← identifier, brand, department, wbs, fluid, createdAt,
                          + все ключи из metadata (по факту встречаемости)
Оборудование
 ├ Система             ← name, category, fileName
 ├ Моноблок            ← name
 └ Элемент             ← name, itemCode, equipType, status, version, updatedAt
    └ Параметры        ← дерево «группа → ключ», у каждого: unit,
                          в скольких элементах встречается, пример значения
Проект                 ← name, code, customer, contractor, description
Справочники            ← по каждому словарю: code, nameRu, иерархия
Закупки                ← этап, даты, статус (из Менеджмента)
Файлы                  ← имя, ревизия, статус, отдел, привязанные теги
Служебное              ← №п/п, сегодняшняя дата, автор, счётчики
```

Детали, без которых будет больно:

1. **Кэш + инвалидация.** Каталог считается за один проход по проекту и кэшируется в
   памяти сервера; сбрасывается по событиям изменения тегов/оборудования (socket.io уже
   есть — те же события, что шлём в интерфейс).
2. **Алиасы параметров.** Один и тот же параметр в разных импортированных бланках
   называется по-разному («Расход воздуха», «Расход, м3/ч», «Производительность»).
   В каталоге показываем частоту встречаемости, а в настройках Конструктора даём
   таблицу алиасов: «считать эти ключи одним полем» (хранить в `AppSetting`,
   ключ `constructor_param_aliases`, на проект). Без этого «колонка на весь проект»
   у половины строк будет пустой.
3. **Учёт `overrides`.** Значение параметра элемента = specs с наложенными
   `overrides` — ровно та же логика, что на экране «Оборудование». Выносим её в общий
   модуль `server/` и используем в обоих местах, чтобы Конструктор никогда не расходился
   с карточкой оборудования.
4. **Единицы измерения** (`unit`) — отдельным свойством поля: мастер сможет вывести их
   в шапку колонки («Высота, мм») или в отдельную колонку.

Поверх каталога — **исполнитель запросов** (`POST /api/constructor/query`):

```jsonc
{
  "projectId": "...",
  "entity": "tag",                     // tag | element | system | file | procurement...
  "filters": [                         // И/ИЛИ-группы
    { "field": "equipment.category", "op": "in", "value": ["AHU", "FAN"] },
    { "field": "department", "op": "contains", "value": "ОВ" }
  ],
  "columns": ["identifier", "brand", "param:Вентилятор|Расход воздуха", ...],
  "sort": [{ "field": "identifier", "dir": "asc" }],
  "groupBy": null                      // или поле — для блоков с подзаголовками
}
```

Ответ — строки + метаданные (id сущностей для каждой строки — нужны для обновления и
для «кликнул по строке → открыл карточку тега»). Один и тот же исполнитель обслуживает
мастера, умные блоки и функции формул — **три фасада над одним механизмом**, а не три
реализации.

## 5. Мастер «Собрать данные» — главный сценарий

Кнопка «Собрать данные» в Библиотеке и в редакторе. Пошаговый визард (в духе
существующих `TagImportWizard`/`DocImportWizard`):

**Шаг 1. Что собираем.** Крупные карточки: «Теги», «Оборудование (элементы)»,
«Системы», «Файлы», «Закупки». Подпись с количеством («247 тегов в проекте»).

**Шаг 2. Какие строки.** Область данных: весь проект / фильтры (конструктор условий
по любому полю каталога: равно, содержит, в списке, пусто/не пусто, диапазон чисел
и дат) / ручной выбор галочками из живого списка. Счётчик «попадает N строк»
обновляется на лету.

**Шаг 3. Какие колонки.** Дерево каталога полей (§4) с чекбоксами и поиском. Выбранные
колонки — справа, перетаскиванием задаётся порядок, у каждой можно переименовать
заголовок и включить «выводить единицу измерения». Рядом с параметрами — заполненность
(«есть у 195 из 247») — сразу видно, где будут дырки.

**Шаг 4. Вид блока.** Направление (строки вниз — стандарт; или транспонировано:
параметры в строках, теги в колонках — частый формат опросных листов), группировка
с подзаголовками (например, по системе), строка итогов (сумма/кол-во по числовым
колонкам), автостиль шапки.

**Шаг 5. Предпросмотр и вставка.** Живой предпросмотр первых ~20 строк. Кнопки:
«Вставить в текущий лист (от активной ячейки)», «На новый лист», «Создать новый
документ». Вставляется **умный блок** (§6), а не мёртвый текст.

Каждый набор шагов можно сохранить как **пресет выборки** («Габариты для заказчика»,
«Кабельный журнал») — пресеты видны на шаге 1, это и есть «множество параметризаций»:
один клик — свежая таблица по актуальным данным.

## 6. Умные блоки: данные, которые живут

Умный блок = диапазон на листе + запрос (§4) + настройки вида. Хранится в `bindings`.

Поведение:

- **Обозначение.** Тонкая цветная рамка/уголок с именем блока (видна в редакторе, не
  печатается). Клик по уголку — панель блока: запрос, колонки, кнопки «Обновить»,
  «Изменить выборку» (снова мастер с текущими настройками), «Отвязать» (сделать
  обычными ячейками), «Удалить блок».
- **Обновление** — только по явной кнопке («Обновить блок» / «Обновить все данные»
  на панели инструментов). Никаких фоновых перезаписей того, на что человек смотрит.
  Перед обновлением — автоверсия (§3).
- **Индикатор устаревания.** При открытии документа сервер отвечает на лёгкий запрос
  «изменилось ли что-то по этим запросам с даты X» (сравнение `updatedAt`) — на блоке
  появляется значок «данные изменились в проекте». Ненавязчиво, но заметно.
- **Ручные правки внутри блока** разрешены. Правка ячейки с привязанным значением
  помечает её как **override** (маленький треугольник-маркер, как в Excel у комментария).
  При обновлении: привязанные ячейки перезаписываются, override-ячейки — нет, но если
  «живое» значение под override изменилось, показываем конфликт (тот же паттерн, что
  `paramConflicts` в оборудовании): списком «ручное значение ↔ новое из проекта»,
  решение по каждой строке или «принять все».
- **Изменение состава строк** при обновлении: новые сущности, попадающие под фильтр,
  дописываются (блок раздвигает лист вниз), исчезнувшие — подсвечиваются и предлагаются
  к удалению. Строки сопоставляются по id сущности, а не по номеру строки — сортировку
  и вставленные пользователем строки-разделители не теряем.
- **Формулы поверх блока** работают как с обычными ячейками: `=СУММ(D2:D40)` по колонке
  веса из блока — обычное дело. После обновления с изменением высоты блока диапазоны
  «итоговых» формул, ссылающихся на блок целиком, корректируем (для этого блок знает
  свои границы).
- **Клик по строке блока** (по номеру-якорю) — переход к сущности: карточка тега,
  элемент оборудования. Дёшево реализуется, сильно повышает полезность.

## 7. Функции формул с данными проекта

Для точечных случаев (шапка документа, одиночное значение в тексте) — свои функции,
регистрируемые в движке. Имена — русские с английскими синонимами, автодополнение
и подсказка синтаксиса в строке формул:

| Функция | Что делает |
|---|---|
| `=ПРОЕКТ("customer")` | поле проекта: name, code, customer, contractor… |
| `=ТЕГ("3700-K02"; "brand")` | поле тега по идентификатору |
| `=ПАРАМ("3700-K02"; "Вентилятор"; "Расход воздуха")` | параметр элемента, привязанного к тегу (с учётом overrides и алиасов) |
| `=ПАРАМ_ЭЛ("бл2.1"; "…"; "…")` | то же по itemCode элемента |
| `=СПРАВ("Отделения"; "K02"; "nameRu")` | значение из справочника |
| `=СЧЁТ_ТЕГОВ("category=AHU")` | количество по мини-фильтру |
| `=СЕГОДНЯ()`, `=ПОЛЬЗОВАТЕЛЬ()` | дата, текущий пользователь |

Как это работает технически: функции **синхронные для движка**, но данные берут из
клиентского кэша проекта (снимок нужных срезов подгружается при открытии документа и
по кнопке «Обновить»). Формула, чьих данных ещё нет в кэше, показывает `#ЗАГРУЗКА`,
докачивает срез и пересчитывается. Так избегаем и подвисаний пересчёта, и тысяч
мелких запросов к серверу.

Плюс полный штатный набор движка: СУММ/SUM, ЕСЛИ/IF, ВПР/VLOOKUP, СЦЕПИТЬ, СЧЁТЕСЛИ,
математика, текст, даты — из коробки.

## 8. Возможности редактора (чек-лист «как настоящий эксель»)

Из коробки движка (проверяем на спайке, включаем):

- несколько листов, переименование/цвет ярлыка;
- шрифты, размер, жирный/курсив/подчёркнутый/зачёркнутый, цвет текста и заливки;
- границы (все варианты), объединение ячеек, выравнивание, перенос текста, поворот;
- числовые форматы (число, %, дата, дробность, свой формат);
- условное форматирование (подсветка по условию, шкалы, полосы данных);
- фильтры и сортировка по диапазону;
- закрепление строк/столбцов, высота/ширина, скрытие;
- формулы + автозаполнение маркером, undo/redo, поиск и замена;
- копирование/вставка: внутри книги, из Excel/в Excel через буфер (TSV/HTML);
- комментарии к ячейкам.

Наши дополнения:

- правая панель **«Данные проекта»**: дерево каталога (§4) — перетащил поле на лист →
  вставилась формула-привязка; перетащил узел «Теги» → предложение открыть мастера;
- вставка изображений (логотип заказчика в шапку) — хранение как data-URI в снапшоте
  с лимитом размера;
- «шапка документа» одним кликом: проект, заказчик, дата, автор (из `=ПРОЕКТ()`/служебных);
- импорт существующего .xlsx как отправной точки (читалка `xlsx` уже в зависимостях;
  честно предупреждаем, что стили при импорте переносятся частично).

Сознательно **не** тащим в первую версию: сводные таблицы, диаграммы, макросы,
одновременное редактирование одного документа. Диаграммы и сводные — кандидаты на
«потом» (см. §13), макросы — никогда (у нас для этого функции и блоки).

## 9. Шаблоны и переносимость

- **«Сохранить как шаблон»** — документ с `kind=TEMPLATE`: структура, стили, формулы,
  блоки с их запросами, но без данных (данные подтянутся при применении).
- **Применение шаблона** к текущему проекту: создаётся документ, все блоки сразу
  обновляются. Поля-алиасы и фильтры, ссылающиеся на отсутствующие в этом проекте
  параметры, показываются списком «не нашлось — сопоставьте или пропустите».
- **Переменные шаблона.** В шаблоне можно объявить параметры (например,
  «Категория оборудования», «Отделение») и использовать их в фильтрах блоков.
  При применении мастер спросит значения. Один шаблон «Габариты для заказчика»
  закрывает и AHU, и вентиляторы, и клапаны.
- **Экспорт/импорт шаблона в файл** (.json) — перенести коллеге на другую машину,
  пока нет общего сервера; при PostgreSQL-режиме шаблоны и так общие.
- Стартовый набор шаблонов кладём в поставку: «Ведомость оборудования», «Габариты и
  веса по тегам», «Опросный лист (параметры в строках)», «Реестр файлов проекта»,
  «Статус закупок» — живая документация возможностей.

## 10. Печать и экспорт

Все экспорты — **серверные** (в Electron сервер локальный, это бесплатно), из снапшота
книги. Один модуль `server/constructorExport.ts` с общей промежуточной моделью
«лист → строки → ячейки со стилями», из неё три бэкенда:

1. **XLSX** — через `exceljs` (MIT): в отличие от community-версии `xlsx`, пишет стили,
   объединения, ширины, форматы чисел, фильтры. Уже установленный `xlsx` остаётся для
   **чтения** (импорт бланков — он и сейчас это делает).
2. **PDF** — рендерим лист в печатный HTML (наш шаблонизатор: таблица + инлайн-стили,
   разбивка на страницы, ориентация/поля из `settings`, колонтитулы: название документа,
   проект, страница N из M) → `webContents.printToPDF` скрытого окна Electron.
   Итог — векторный, точный PDF без внешних зависимостей. В dev-режиме без Electron —
   фоллбэк `window.print()` этого же HTML.
3. **DOCX** — библиотека `docx` (MIT): таблица со стилями, шапка документа, колонтитулы.
   Word-таблица ≠ Excel-сетка, поэтому экспортируем «печатную область» (как PDF),
   без формул — честно пишем это в подсказке.

**Печать** — тот же HTML-рендер, что и PDF, через диалог печати Electron. Предпросмотр
печати — обязателен (модалка с этим HTML и настройками: ориентация, масштаб
«вписать по ширине», повторять строку шапки на каждой странице, поля).

**Куда сохранять** — единый диалог «Экспорт»:
- **в Проводник программы**: создаётся `FileNode` (`filePath: /shared/...` или личный
  раздел, статус «D», ревизия «1») в выбранной папке — документ сразу живёт в
  документообороте, с тегами и статусами как любой файл;
- **на диск** — стандартный диалог сохранения;
- и туда, и туда одновременно.

## 11. API сервера

```
GET    /api/constructor/docs?projectId=&kind=      список документов/шаблонов
POST   /api/constructor/docs                        создать (пустой | из шаблона | из мастера)
GET    /api/constructor/docs/:id                    документ целиком
PUT    /api/constructor/docs/:id                    автосейв (workbook/bindings/settings/name)
DELETE /api/constructor/docs/:id
POST   /api/constructor/docs/:id/duplicate
GET    /api/constructor/docs/:id/versions           список версий
POST   /api/constructor/docs/:id/versions           ручная версия
POST   /api/constructor/docs/:id/restore/:version   откат (с автоверсией текущего)

GET    /api/constructor/catalog?projectId=          каталог полей (§4)
POST   /api/constructor/query                       исполнить запрос (§4)
POST   /api/constructor/query/check-fresh           «изменилось ли» для блоков документа

POST   /api/constructor/export                      { docId, format: xlsx|pdf|docx,
                                                      target: download|explorer|both,
                                                      folderId?, scope?, printSettings? }
```

Роуты — отдельным файлом `server/routes/constructor.ts` через общий контекст
(рефакторинг под это уже сделан). Изменения документов пишем в `SystemChangeLog`
(«Создан документ Конструктора “Габариты для заказчика”», `targetRoute:
/constructor?doc=...`) и шлём событие в socket.io — список у коллег обновится.

## 12. Права, конкуренция, поведение

- **Права** — в существующую систему `permissions`: `constructor.view`,
  `constructor.edit`, `constructor.export`, `constructor.templates` (управление общими
  шаблонами). По умолчанию инженерам всё, кроме управления общими шаблонами.
- **Одновременное редактирование**: без CRDT-героики. Мягкая блокировка: при открытии
  редактора документ помечается «редактирует Иванов» (socket.io, TTL с heartbeat);
  второй пользователь открывает в режиме чтения с кнопкой «Перехватить» (первому —
  уведомление). Автосейв + версии делают гонки неопасными.
- **Большие объёмы**: блок на несколько тысяч строк — нормальный случай (виртуализация
  в движке есть). Запросы исполняются на сервере, на клиент едут готовые строки.
  Лимит-предохранитель на вставку (например, 50 000 ячеек за раз) с честным
  предупреждением.
- **Удаление сущностей проекта**: блок при обновлении сам покажет исчезнувшие строки
  (§6) — ссылочной целостности с ячейками специально не строим, документ обязан
  оставаться читаемым даже если полпроекта удалили.
- **ИИ-ассистент** (задел): намерение «собери таблицу: все AHU, высота, ширина, вес» →
  ассистент открывает мастера с предзаполненными шагами. Каталог полей (§4) — готовая
  основа для такого разбора фразы.

## 13. Типовые сценарии (проверочный список для приёмки)

1. Таблица для заказчика: все теги проекта × габариты/вес/электрика — мастером за
   5 шагов, шапка с логотипом, PDF в Проводник.
2. Опросный лист: параметры в строках, теги в колонках (транспонированный блок).
3. Спецификация по системе: блок с группировкой по `EquipmentSystem`, итоги по группам.
4. Ведомость закупок: блок из Менеджмента + условное форматирование просроченных.
5. Реестр выданных документов: блок по файлам Проводника (ревизия, статус, отдел).
6. Сводка руководителю: лист с одиночными формулами (`=СЧЁТ_ТЕГОВ`, `=ПРОЕКТ`) без блоков.
7. Один шаблон «Габариты» с переменной «категория» → три документа: AHU, вентиляторы, клапаны.
8. Обновление после импорта новой ревизии бланка: открыл документ → значок «данные
   изменились» → «Обновить все» → конфликты с ручными правками разрешены поштучно →
   экспорт новой ревизии в ту же папку Проводника.
9. Инженер вставил свои строки-примечания внутрь блока и покрасил их — обновление
   данных их не тронуло.
10. Перенос шаблона в другой проект, где параметр называется иначе, — мастер алиасов
    предложил сопоставление.

## 14. Риски

| Риск | Смягчение |
|---|---|
| Движок: платные модули/смена формата снапшота | спайк до старта; версия запинена; наш слой (блоки, функции, экспорт) от снапшота изолирован адаптером |
| Разнобой имён параметров → пустые колонки | алиасы (§4.2) + показ заполненности на шаге выбора колонок |
| «Обновить» испортило ручную работу | автоверсия перед обновлением + overrides + разрешение конфликтов |
| Размер бандла | ленивый route-чанк; на старте приложения Конструктор не грузится |
| Ожидание «прям как Excel» у сводных/диаграмм | честные рамки в UI; сводные/диаграммы — отдельной фазой |
| Экспорт «не совпал с экраном» | единая промежуточная модель для XLSX/PDF/печати; приёмка по сценариям §13 |

## 15. План внедрения по фазам

- **Фаза 0 — спайк (1–2 дня).** Univer в ленивом чанке: снапшот в SQLite, кастомная
  функция, вставка 5 000 строк, копипаста из Excel. Решение «да/фоллбэк».
- **Фаза 1 — MVP.** Раздел + библиотека, редактор (штатные возможности движка),
  автосейв, каталог полей, мастер «Собрать данные» со вставкой **статичной** таблицы
  (ещё без живых блоков), экспорт XLSX, сохранение в Проводник.
- **Фаза 2 — живые данные.** Умные блоки: обновление, overrides, конфликты, индикатор
  устаревания, версии документа, клик-переход к сущности.
- **Фаза 3 — функции и шаблоны.** `=ТЕГ()/ПАРАМ()/ПРОЕКТ()...`, панель «Данные
  проекта» с drag&drop, шаблоны с переменными, стартовый набор шаблонов, алиасы.
- **Фаза 4 — печать и остальной экспорт.** Печатный HTML-рендер, предпросмотр, PDF,
  DOCX, колонтитулы, права, блокировка «кто редактирует».
- **Фаза 5 — полировка.** Транспонированные блоки, группировки/итоги, условное
  форматирование из мастера, интеграция с ИИ-ассистентом, диаграммы/сводные — по спросу.

Каждая фаза самоценна: уже после Фазы 1 инженер собирает таблицу для заказчика и
выгружает её в Excel — дальше документ только «оживает».
