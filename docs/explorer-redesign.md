# Проводник 2.0: пересмотр функций, вида и багов

Проектирование (без применения). Цель — сделать Проводник **центром документооборота
проекта**, а не просто списком файлов: чтобы файл жил в связке с тегами, оборудованием,
закупками и Конструктором; чтобы частые действия делались в один клик; чтобы вид был
современным и одинаково удобным инженеру, менеджеру и руководителю. Документ опирается
на разбор `src/screens/Explorer.tsx` (1961 строка) и файловых роутов `server.ts`.

---

## 1. Что есть сейчас (сильные стороны — сохранить)

- **Виртуальные разделы** «Общий» / «Личный» (у каждого пользователя свой), у админа —
  чужие личные; scope наследуется рекурсивно при переносе;
- список и плитка, виртуализация (`@tanstack/react-virtual`) — тянет тысячи файлов;
- выделение (клик/Shift/Ctrl), контекстное меню, копировать/вырезать/вставить,
  drag&drop загрузка и перенос, горячие клавиши (Ctrl+C/X/V, Delete);
- панель предпросмотра (картинки/PDF/текст), свойства с ревизией/статусом/отделом;
- интеграция: «В оборудование…» с предпросмотром импорта (Фаза 2), зеркало документов
  Конструктора (`type=CONSTRUCTOR`, двойной клик открывает документ);
- статус-бейдж (A/B/C/D) и системные папки.

Фундамент крепкий. Проблемы — в отделке, целостности данных и «недокрученных» функциях.

## 2. Баги и дефекты (найдено в коде)

| # | Где | Проблема | Последствие |
|---|---|---|---|
| B1 | `handleItemDoubleClick` | двойной клик по **обычному** файлу не делает ничего (открывает только папки и зеркала Конструктора) | главный жест проводника мёртв: чтобы увидеть файл, нужно вручную открыть панель предпросмотра |
| B2 | `uploadFiles`: `size: file.size \|\| Math.floor(Math.random()*5000000)` | при нулевом размере пишется **случайный** размер | ложные размеры в БД, «7.0 MB» у пустого файла |
| B3 | `uploadFiles`: `if (file.size < 5MB)` иначе `content=undefined` | файлы > 5 МБ загружаются **без содержимого, молча** | их нельзя ни скачать, ни открыть; пользователь не предупреждён |
| B4 | `filePath: /shared/${file.name}` всегда | путь захардкожен, даже для личных | вводит в заблуждение; при копировании файла имя-коллизия не разрешается (дедуп только в `createEmptyFile`) |
| B5 | предпросмотр PDF: `<iframe src={dataURI} sandbox="">` | `sandbox=""` запрещает всё, включая скрипты встроенного PDF-вьюера | PDF часто не рендерится — пустой фрейм |
| B6 | `POST /api/files`: `data = {...req.body}` → `create({data})` | массовое присваивание любых полей клиента | риск записи произвольных полей; `createdBy` как объект-связь может падать |
| B7 | контент — base64 в колонке БД (`FileNode.content`) | каждый файл целиком в SQLite/Postgres | раздувание БД и трафика (каждое скачивание тащит base64 через JSON); из аудита онлайна |
| B8 | загрузка drag&drop не дедупит имена | два файла с одним именем в папке | путаница, а сопоставление с оборудованием идёт **по имени файла** (`fileName`) → конфликты импорта |
| B9 | отделы захардкожены (`КИПиА/ОВиК/Менеджеры`) | список в коде | не совпадает с реальными отделами компании |
| B10 | нет обработки ошибок загрузки | `await fetch(...)` без проверки `res.ok` в цикле | файл «загрузился» в UI, но упал на сервере — тост «успешно» лжёт |

## 3. Пробелы функционала (с учётом всей программы)

Проводник изолирован от того, ради чего существует программа. Чего не хватает:

1. **Ревизии и версии файла.** Поля `revision`/`statusCode` есть, но версии не хранятся:
   загрузил новую версию чертежа — старая потеряна. Нет истории «кто, когда, что».
2. **Связь файла с сущностями проекта** видна, но односторонняя: у файла есть
   main/additional-теги, но нельзя от **тега/оборудования** прыгнуть к его файлам.
3. **Статусы документооборота** (A/B/C/D) не объяснены и не управляются потоком
   (согласование, «выдан», «в работе») — просто цветная точка.
4. **Нет «недавних» и «избранного»** — при сотнях файлов каждый раз кликать по дереву.
5. **Поиск только по имени** и без фильтров (тип, статус, отдел, тег, дата, автор).
6. **Массовые операции бедные**: нельзя разом сменить статус/отдел/теги выделенным.
7. **Нет предпросмотра Office** (docx/xlsx) — только картинки/PDF/текст.
8. **Нет колонки «связан с»** (тег/оборудование/закупка) — файл-сирота неотличим от
   привязанного к делу.
9. **Права на уровне файла** отсутствуют: личное/общее — единственная граница.
10. **Нет журнала действий по файлу** (кто скачал/переименовал/сменил ревизию).

---

## 4. Редизайн: принципы

1. **Файл — узел графа проекта, а не строка.** У каждого файла — связи (теги,
   оборудование, закупки, документ Конструктора) и статус в потоке. Проводник их
   показывает и даёт переходить в обе стороны.
2. **Один клик до частого.** Открыть, скачать, сменить статус, привязать тег — без
   вложенных диалогов. Двойной клик **открывает** (просмотр), не «ничего».
3. **Современный вид, обе темы.** Плотный, спокойный, с быстрым предпросмотром;
   вид под задачу (список для работы, крупная плитка для чертежей/картинок).
4. **Данные честные.** Реальные размеры, реальные пути, дедуп имён, предупреждения о
   лимитах — никаких random-заглушек.
5. **Будущее без переусложнения.** Ревизии, потоки статусов, умный поиск — но каждый
   через минимум действий и с разумными умолчаниями.

## 5. Вид и взаимодействие (новый макет)

```
┌───────────────────────────────────────────────────────────────────────┐
│  ← →  ↑   Общий / Проектная документация / Чертежи      [⌗ поиск+фильтр] │  ← адрес-строка + умный поиск
├──────────┬────────────────────────────────────────────────┬───────────┤
│ Дерево   │  Панель инструментов (контекстная к выделению)   │  Инспектор│
│ разделов │  [+ Создать][↑ Загрузить][вид][сортировка]       │  выбранного│
│          │                                                  │  файла:   │
│ ⭐ Избран.│  Имя           Статус  Ревиз.  Связь   Изменён   │  превью,  │
│ 🕘 Недавн.│  ▸ Чертежи…                                       │  связи,   │
│ 📁 Общий  │  📄 План П1     ● Выдан  Рев.3  🏷AHU-2  вчера    │  ревизии, │
│  └ …      │  📄 Опросник    ○ D      Рев.1  —       3 дня     │  действия │
│ 🔒 Личный │  🖼 Схема.png   ● B      Рев.1  ⚙ вент.  сегодня  │           │
└──────────┴────────────────────────────────────────────────┴───────────┘
```

### 5.1. Дерево слева — с быстрыми входами
- **⭐ Избранное** и **🕘 Недавние** сверху (частая нужда §3.4) — личные, в localStorage
  или `AppSetting`.
- Разделы «Общий»/«Личный» как сейчас; папки раскрываются инлайн; drag&drop между ними.
- у папки — счётчик файлов и, при наведении, «+» (создать внутри) и «↑» (загрузить сюда).

### 5.2. Умный поиск + фильтры (одна строка)
- поиск по имени **и** по связям (тег, код оборудования): «AHU-2» находит и файлы с
  этим тегом;
- чипы-фильтры рядом: Тип · Статус · Отдел · Тег · Дата · Автор — по клику; активные
  видны и снимаются крестиком;
- поиск **рекурсивный** по разделу (сейчас — по разделу, но без фильтров);
- сохранённые запросы («Мои выданные чертежи») — как пресеты Конструктора.

### 5.3. Список — новые колонки
- **Статус** (цветной чип с названием, не безымянная точка) — клик открывает смену;
- **Ревизия** — клик открывает историю версий;
- **Связь** — чип тега/оборудования/закупки/документа; клик ведёт к сущности;
- сортировка по любой; выбор видимых колонок (личная настройка).

### 5.4. Инспектор справа (заменяет узкую панель предпросмотра)
- крупный предпросмотр (картинки/PDF — **исправить sandbox** §B5; docx/xlsx — рендер
  первой страницы через существующие извлекатели `src/import`);
- секции: **Связи** (теги/оборудование/закупки — добавить/убрать здесь же),
  **Ревизии** (список версий, «загрузить новую версию», откат), **Свойства**
  (отдел, статус, автор, размер), **Действия** (скачать, открыть, в оборудование,
  переименовать, удалить);
- всё inline — без модалок для частого.

### 5.5. Двойной клик = открыть (B1)
- картинка/PDF/текст → инспектор в режиме «крупный просмотр» (или системное окно);
- docx/xlsx → предпросмотр + кнопка «скачать/открыть в приложении»;
- зеркало Конструктора → документ (как сейчас);
- папка → вход. Никогда «ничего не происходит».

### 5.6. Современная подача
- спокойная плотность, крупные иконки типов, цветовые акценты статусов из палитры
  программы (emerald), полноценные светлая/тёмная темы;
- плавные переходы (motion уже есть), скелетоны вместо пустоты при загрузке;
- контекстное меню сгруппировано (Открыть/Скачать · Связи/Статус · Правка · Опасное).

## 6. Функции-«ноу-хау» (будущее, но без лишних действий)

1. **Ревизии файла из коробки.** «Загрузить новую версию» поверх файла: старая уходит
   в историю, ревизия +1, статус можно поднять. Откат в один клик. Модель:
   `FileRevision { fileId, revision, content/ref, statusCode, authorId, note, createdAt }`.
2. **Поток статусов документа.** A/B/C/D → человекочитаемо (Черновик → На проверке →
   Согласован → Выдан) с настраиваемыми в «Настройках» названиями/цветами (как этапы
   закупок). Смена статуса — из чипа, пишется в лог и уведомляет заинтересованных.
3. **Двусторонние связи.** На карточке тега/оборудования — вкладка «Файлы»; в Проводнике
   — колонка «Связь». Привязка тянет за собой: файл, привязанный к тегу AHU-2, виден и
   там, и там; ИИ-ассистент «покажи чертежи AHU-2» отвечает из этого.
4. **Умное авто-предложение связи** при загрузке: если имя файла содержит код тега/
   оборудования проекта — предложить привязку одним кликом (не навязывая).
5. **Пакетные действия** над выделением: статус, отдел, теги, «в оборудование»,
   «скачать архивом» — панель появляется при мультивыборе.
6. **Быстрый обмен**: «поделиться в чат» (уже есть share-механизм) и «прикрепить к
   закупке» прямо из контекста.
7. **Скачать папку/выделенное ZIP-архивом** (серверная сборка, как экспорт Конструктора).
8. **Пометки и цвета** файлов/папок (личные), «избранное».
9. **Дубли и потерянные**: индикатор «файл ни с чем не связан» и «дубликат имени» —
   помогает навести порядок (перекликается с поиском дублей оборудования).
10. **Печать/PDF-титул**: собрать реестр выданных документов проекта — по сути готовый
    сценарий Конструктора (блок по файлам), кнопка «Реестр в Конструктор».

## 7. Целостность данных и хранилище (чинит B2–B8)

1. **Реальные размеры**: писать `file.size` как есть; 0 — это 0, не random (B2).
2. **Лимит контента честно**: файлы > порога — предупреждение «превысит лимит
   предпросмотра, хранить только метаданные?» или (лучше) файловое хранилище (п.4).
3. **Пути осмысленные**: `filePath` = раздел + путь папки + имя; личные — свой префикс (B4).
4. **Файлы на диск сервера** (как вложения чата, `/chat_files` уже так): в БД —
   метаданные + ссылка, контент — файлом; скачивание по URL, не base64-через-JSON (B7).
   Это же снимает лимит 5 МБ и раздувание БД. Отдельная фаза (совпадает с «файлы-2»
   из online-readiness-audit.md §5.3).
5. **Дедуп имён при любой загрузке/копировании** (B8) — вынести из `createEmptyFile`
   в общий помощник; сопоставление с оборудованием по стабильному id, не по имени.
6. **`POST /api/files` — белый список полей** (B6): принимать только name/folderId/
   scope/ownerId/type/size/content/department/status/revision, не весь body.
7. **Проверка `res.ok`** в цикле загрузки (B10): собирать успехи/ошибки, тост честный
   («загружено 3, ошибка у 1»).

## 8. Права и роли (разные стороны использования)

- **Инженер**: свои личные + общие; создаёт, грузит версии, ставит статусы своих
  документов, привязывает теги.
- **Менеджер/руководитель**: видит статусы потоком, реестры выданного, фильтры «что
  не согласовано»; управляет общими статусами/отделами.
- **Админ**: чужие личные (как сейчас), настройка статусов/отделов, передача личных
  файлов при увольнении (перекликается с Конструктором §III.8).
- Действия уважают авторизацию API (шаг 4 онлайна уже внедрён) — личное фильтруется по
  токену, а не по query-параметру.

## 9. Фазы внедрения (каждая самоценна, минимум действий у пользователя)

| Фаза | Содержание | Эффект |
|---|---|---|
| **1. Гигиена и баги** | двойной клик = открыть (B1), реальные размеры (B2), предупреждение о лимите (B3), sandbox PDF (B5), дедуп имён (B8), белый список полей (B6), честные тосты (B10) | сразу перестаёт врать и «молчать»; главный жест оживает |
| **2. Современный вид** | новый макет (дерево с Избранным/Недавним, инспектор справа, колонки Статус/Ревизия/Связь), обе темы, скелетоны, сгруппированное меню | вид «будущее», меньше кликов |
| **3. Умный поиск + фильтры** | рекурсивный поиск по имени и связям, чипы-фильтры, сохранённые запросы | найти нужное за секунды |
| **4. Связи и статусы** | двусторонние связи файл↔тег/оборудование/закупка, поток статусов с настройкой, авто-предложение связи при загрузке, пакетные действия | Проводник становится документооборотом |
| **5. Ревизии** | версии файла, «новая версия», история, откат | конец «перезаписал — потерял» |
| **6. Хранилище на диске** | файлы на диск сервера + метаданные в БД, скачивание по URL, ZIP-архивом, снятие лимита 5 МБ | масштаб и скорость (совместно с online-audit §5.3) |

Фаза 1 — быстрая, снимает боль и ложь данных. Фазы 2–3 — вид и поиск. 4–5 — превращают
в документооборот. 6 — инфраструктура.

## 10. Отвергнутое (чтобы не возвращаться)

| Идея | Почему нет |
|---|---|
| Полноценная файловая система с симлинками/ярлыками | путаница «где оригинал»; у файла одно расположение (как решили в Конструкторе §III.5) |
| Онлайн-редактирование docx/xlsx в Проводнике | это работа Конструктора и внешних приложений; Проводник — хранение и связи |
| Автоимпорт всех загруженных файлов в оборудование | уже отвергнуто: импорт только вручную с предпросмотром (Фаза 2 импорта) |
| Произвольные вложенные права на каждый файл | сложность; scope + статусы + авторство покрывают реальные сценарии отдела |
