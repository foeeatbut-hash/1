# Конструктор, часть IV: совместное онлайн-редактирование

Продолжение частей I–III. Пересматривает решение части II §10 (блокировка «пишет один»):
программа работает и как сетевая (общая PostgreSQL, socket.io уже в основе), поэтому
несколько человек должны редактировать один документ **одновременно**, как в Google
Sheets. Часть II §16.4 отвергала CRDT — это остаётся в силе: полноценный CRDT не нужен,
достаточно **серверной сериализации операций с трансформацией (OT-lite)** — на порядок
проще, потому что сервер один и он локален/общий, офлайн-ветвления истории нет.

Два режима работы БД → два поведения, переключаются сами, без настроек:

| Режим | Поведение редактора |
|---|---|
| LOCAL (SQLite, один пользователь) | совместность не нужна: движок работает напрямую, op-лог пишется тем же кодом (журнал для undo/версий), сетевой слой выключен |
| REMOTE (PostgreSQL, много пользователей) | полноценное совместное редактирование, описанное ниже |

---

## 1. Архитектура: сервер — единственный судья порядка

Всё редактирование в движке — команды через командную шину (часть II §1.4 уже
перехватывает их для `bindingSync`). Тот же перехват — транспорт коллаборации:

```
клиент А: команда → применяется ЛОКАЛЬНО сразу (optimistic) →
          op {docId, baseRev, clientOpId, payload} → socket.io → сервер

сервер:   очередь операций документа (строго последовательная) →
          трансформация op против всех op с rev > baseRev (§3) →
          присвоение rev = last+1 → запись в op-лог → рассылка всем в комнате doc:<id>

клиент А: получает свой op с rev → подтверждён (ack), убирает из очереди неотправленного
клиент Б: получает чужой op → трансформирует против своих неподтверждённых → применяет
          как «немую» команду (без записи в undo-стек Б и без повторной отправки)
```

Ключевые свойства:

- **ноль ожидания при вводе**: свои правки применяются мгновенно, сеть — вдогонку;
- **сходимость**: порядок один для всех (серверный rev), трансформация двусторонняя
  (клиент трансформирует чужое против своего неподтверждённого, сервер — входящее
  против уже принятого) — классическая схема клиент-серверного OT;
- **одна очередь на документ** на сервере — никакой конкурентности внутри документа,
  вся сложность OT сводится к матрице §3.

## 2. Модель операций

Операция — не «снапшот-дифф», а семантическая единица, совпадающая с командами движка:

```
адресные:     setCells (значения+стили пачкой), setNumberFormat, setComment
структурные:  insertRows, deleteRows, insertCols, deleteCols,
              setRowHeight/ColWidth, merge, unmerge,
              addSheet, removeSheet, renameSheet, moveRange (вырезать-вставить)
блочные:      blockRefresh, blockEditQuery, blockDetach, blockDelete (§5)
документные:  renameDoc, setScope, setFolder, addVersion
```

`bindings` (часть II §1) меняются **как часть той же операции** (payload несёт и дельту
привязок) — лист и привязки не могут разъехаться из-за сети, потому что едут вместе.

Гранулярность: непрерывный ввод в одну ячейку коалесцируется в один op при коммите
ячейки (Enter/blur) — не шлём каждый символ; выделение-и-заливка 500 ячеек — один op.

---

## 3. Матрица трансформаций (OT-lite)

Обозримая, потому что типов мало. Правила `transform(текущий, против)`:

| текущий \ против | адресный | insert/delete Rows/Cols | merge | moveRange |
|---|---|---|---|---|
| **адресный** | той же ячейки? → LWW: серверный порядок побеждает, проигравший просто перекрыт (как в Google Sheets); разных — не мешают | сдвиг адресов на вставленное/удалённое; адрес внутри удалённого диапазона → op отбрасывается (ячейки больше нет) | адрес внутри объединения → переадресация в якорную ячейку | сдвиг адресов вслед за переносом |
| **insert/delete** | — (не влияет) | сдвиг позиции вставки/границ удаления; пересечение двух delete → вычитание диапазонов | не мешают | сдвиг позиций |
| **merge/unmerge** | не мешают | сдвиг диапазона объединения; частично удалён → op отбрасывается | пересекающиеся merge → второй отбрасывается + тост автору | сдвиг |
| **блочные** | см. §5 | см. §5 | внутри блока merge запрещён (часть II §1.4) | перенос блока = документная op |

Отброшенная операция — не ошибка: автору тост «строка была удалена коллегой», его
локальное состояние чинится обратным применением (инверсия хранится в undo-стеке).

Инвариант тестируется фаззером (§9): при любой паре конкурирующих операций все клиенты
сходятся к байт-идентичному снапшоту.

## 4. Присутствие (presence)

Отдельный лёгкий канал (не пишется в op-лог, шлётся с троттлингом 150 мс):

- **кто в документе**: аватарки-кружки в шапке редактора («+3»), цвет каждого —
  стабильный хэш userId (как цвета групп в чате);
- **чужой курсор/выделение**: цветная рамка с флажком-именем; при редактировании
  ячейки — пунктирная рамка «Иванов вводит…» (сам ввод не транслируется до коммита);
- **режим просмотра тоже виден**: глазок у аватарки — «смотрит, не редактирует»;
- клик по аватарке — прыжок к выделению коллеги (разовый, не «слежение» — follow-режим
  в отвергнутое: дорого, просят редко).

Presence живёт в памяти сервера (socket.io-комната), исчезает с дисконнектом — БД
не трогаем.

---

## 5. Умные блоки при коллаборации

Обновление блока (сверка части II §2) трансформации не поддаётся — это крупная
составная операция. Решение — **микро-монополия на диапазон**:

```
клиент жмёт «Обновить блок» → сервер ставит blockRefresh в очередь документа →
на время исполнения (секунды) операции ТРЕТЬИХ лиц, адресованные в диапазон блока,
отбрасываются с тостом «блок обновляется (Иванов)» → результат сверки рассылается
одним op → монополия снята.
```

- у остальных на блоке — синяя пульсация «обновляется…» (presence-канал);
- конфликты overrides (часть II §2.6) разрешает **инициатор обновления**; его решения
  едут теми же op'ами;
- «Изменить выборку» (мастер) — так же, через монополию на время применения;
- одновременный «Обновить» двух людей → второй получает «уже обновляется» и результат
  первого (не очередь из двух сверок подряд).

## 6. Хранение: op-лог + компакция

```prisma
model ConstructorOp {
  id        String   @id @default(uuid())
  docId     String
  rev       Int      // порядковый номер в документе, уникален в паре (docId, rev)
  authorId  String?
  payload   String   // JSON операции (+дельта bindings)
  createdAt DateTime @default(now())

  @@unique([docId, rev])
}
```

- **Снапшот-компакция**: каждые 200 операций или 60 с тишины сервер материализует
  текущее состояние в `ConstructorDoc.workbook`/`bindings` (это и есть «автосейв» —
  клиентский PUT из части II §7.2 в REMOTE-режиме не нужен) и удаляет op'ы старше
  последнего снапшота версии;
- **версии** (часть I §3) — снимки поверх той же механики: автоверсия перед
  blockRefresh, ручные, ретеншн без изменений. Восстановление версии = документная
  операция через ту же очередь (все участники переезжают синхронно, с подтверждением
  «Иванов восстанавливает версию от …» у остальных);
- **подключение к документу**: клиент получает снапшот + хвост op'ов после него,
  применяет, входит в комнату. Ничего не блокируется.

## 7. Обрывы связи и офлайн

- Дисконнект → баннер «нет связи, изменения не отправляются», ввод разрешён,
  неподтверждённые op'ы копятся в очереди (и дублируются в localStorage — часть II
  §7.3 переиспользуется);
- реконнект → клиент шлёт очередь с исходным baseRev, сервер трансформирует как
  обычно — чаще всего всё просто вливается;
- лимит очереди (500 op'ов / 5 минут) превышен → ввод замораживается («слишком долго
  без связи»), после реконнекта: если op-лог сервера уже обрезан компакцией ниже
  baseRev — полная перезагрузка снапшота, свои неподтверждённые применяются поверх,
  неприменимые показываются списком «не удалось применить» (редкий, честный исход);
- падение сервера → op-лог в БД, компакция при старте доигрывает хвост — потери нет.

## 8. Undo/redo при коллаборации

- Undo — **персональный**: отменяет последнюю СВОЮ операцию (стандарт Google Sheets);
- инверсия операции хранится в локальном undo-стеке; при отмене инверсия
  трансформируется против всех операций, пришедших после оригинала, и уходит на сервер
  обычным op'ом;
- если после чужих структурных правок инверсия неприменима (строку удалили) —
  честный тост «отменить уже нельзя», шаг пропускается;
- redo симметричен. Чужие операции в мой undo-стек не попадают никогда.

## 9. Тестирование сходимости

1. **Фаззер**: N виртуальных клиентов, случайные операции из §2, случайные задержки и
   реконнекты, в конце — снапшоты всех клиентов и сервера байт-идентичны. Гоняется в CI,
   сид пишется в лог для воспроизведения.
2. **Юнит на каждую клетку матрицы §3** — вручную построенные пары операций.
3. **Сценарные**: двое одновременно в одном блоке; обновление блока под чужой ввод;
   восстановление версии при активной сессии; отвал сети на 4 минуты с правками.

## 11. Бюджеты и лимиты

| Параметр | Значение |
|---|---|
| участников в документе | до 10 одновременно (мягкий лимит с предупреждением) |
| пропускная способность | 20 op/с на документ — очередь последовательная, укладывается с запасом |
| задержка эха (op туда-обратно, LAN) | < 100 мс |
| хвост op'ов при подключении | < 200 (иначе сначала компакция) |
| presence-трафик | ≤ 7 сообщений/с на клиента (троттлинг 150 мс) |

## 12. Поэтапность

Коллаборация — отдельная фаза после Фазы 2 (живые блоки), **но закладывается с Фазы 1**:
все правки уже идут командами через перехват (нужно для bindingSync), op-лог включается
в LOCAL-режиме как журнал undo/восстановления — то есть к моменту фазы коллаборации
остаётся добавить сетевой транспорт, трансформации и presence, не переписывая редактор.
MVP может жить с блокировкой части II §10 — форматы совместимы, переключение прозрачно.
