# Конструктор, часть II: глубокая проработка подсистем

Продолжение `constructor-design-v0.25.md` (часть I — обзор и решения). Здесь — уровень,
по которому можно писать код: форматы данных, алгоритмы, грамматики, крайние случаи,
лимиты, тест-план. Нумерация разделов своя; ссылки «§I.n» ведут в часть I.

---

## 1. Формальная модель привязок (`bindings`)

`ConstructorDoc.bindings` — JSON с версией схемы (мигрируем явно, как снапшоты движка):

```jsonc
{
  "schemaVersion": 1,
  "blocks": [ /* умные блоки */ ],
  "cellFunctions": { /* служебный кэш: какие срезы данных нужны формулам, §6.4 */ }
}
```

### 1.1. Анатомия блока

```jsonc
{
  "id": "blk_8f3a",                  // стабильный id, живёт всю жизнь блока
  "name": "Габариты AHU",            // человеческое имя (уголок, панель блока)
  "sheetId": "sh_01",
  "anchor": { "row": 4, "col": 1 },  // верхний-левый угол ОБЛАСТИ ДАННЫХ (не шапки)
  "headerRows": 1,                   // сколько строк шапки принадлежит блоку (0..2)
  "totalsRows": 1,                   // строк итогов снизу (0..1 на блок + по группам)
  "layout": "rows" | "transposed",
  "query": { /* §I.4: entity, filters, sort, groupBy */ },
  "columns": [ /* §1.2 */ ],
  "rows": [ /* §1.3 — реестр строк области данных, по порядку */ ],
  "overrides": { "r:tag:uuid|c:col_h": {"user":"...","at":"...","value":"…"} },
  "conflicts": [ /* нерешённые после последнего обновления, §2.6 */ ],
  "style": { "headerPreset": "slate", "banding": true },
  "state": { "lastRefreshAt": "...", "lastRefreshBy": "...", "stale": false }
}
```

Координата хранится **одна** — якорь; всё остальное (высота, ширина) выводится из
`rows`/`columns`. Так нет двух источников правды, которые могут разъехаться.

### 1.2. Колонки

```jsonc
{ "id": "col_h", "kind": "field",   "path": "param:Габариты|Высота", "title": "Высота, мм", "withUnit": false }
{ "id": "col_n", "kind": "rownum" }                                  // №п/п, перенумеровывается при обновлении
{ "id": "col_v", "kind": "formula", "template": "={{col_h}}*{{col_w}}/1e6", "title": "Площадь, м²" }
{ "id": "col_s", "kind": "static" }                                  // колонка «для пометок»: данные не пишем, обновление не трогает
```

`kind:"formula"` — **вычисляемая колонка**: шаблон с плейсхолдерами `{{col_id}}`,
при заполнении строки плейсхолдеры разворачиваются в относительные A1-ссылки этой же
строки. При обновлении блока формула перезаливается в новые строки и не трогается в
строках, где пользователь её переписал (это становится override ячейки). Это закрывает
сценарии «объём = В×Ш×Г», «цена × кол-во», «статус по ЕСЛИ()» без серверных выражений.

### 1.3. Реестр строк

`rows` — массив, параллельный физическим строкам области данных:

```jsonc
[
  { "key": "tag:6f2c…" },                 // строка данных: entityKey = "<entity>:<id>"
  { "key": "el:9a11…" },
  { "free": true },                       // строка, вставленная пользователем ВНУТРИ блока
  { "group": "Система У1" },              // строка-подзаголовок группировки
  { "key": "tag:5b90…", "missing": true } // сущность исчезла из выборки (§2.4)
]
```

`entityKey` — а не номер строки — основа сверки при обновлении (§2). Свободные строки
(`free`) обновление обходит стороной: примечания, разделители, покрашенные строки
инженера переживают любые обновления (§I.13, сценарий 9).

### 1.4. Поддержание реестра при правках листа

Единственный надёжный способ не разъехаться с движком — **перехват его команд**
(у Univer командная шина: каждое действие, включая undo/redo, — команда). Пишем один
модуль-синхронизатор `bindingSync`, подписанный на команды листа. Таблица семантики:

| Действие пользователя | Реакция синхронизатора |
|---|---|
| Вставка строки внутри области данных | вставить `{free:true}` в ту же позицию `rows` |
| Вставка строки выше блока / вставка столбца левее | сдвинуть `anchor` |
| Удаление строки данных | удалить запись из `rows` (сущность вернётся при обновлении, если всё ещё проходит фильтр — предупреждаем тостом) |
| Удаление строк, пересекающих границу блока | как обычное удаление: часть из `rows`, часть — строки листа вне блока |
| Правка ячейки данных | записать override (§2.5) + маркер-уголок |
| Правка ячейки free-строки/`static`-колонки | ничего (обычная ячейка) |
| Вырезать-вставить весь блок (диапазон ⊇ блок) | перенести `anchor`/`sheetId` |
| Вырезать-вставить кусок блока | перенесённые ячейки отвязываются (становятся обычными), в блоке — overrides «пусто»; предупреждение |
| Копировать блок → вставить | вставляется **значениями** (без привязки); «Дублировать блок» — отдельная команда в панели блока |
| Объединение ячеек внутри области данных | запрещаем с пояснением (ломает сверку); в шапке/итогах — можно |
| Удаление листа с блоком | подтверждение «на листе N живых блоков» |
| Сортировка/фильтр средствами листа по области блока | перехватываем и выполняем как изменение `query.sort`/`filters` блока — иначе порядок `rows` и листа разойдётся |
| Undo/redo | команды идут через ту же шину — синхронизатор видит обратные операции; `bindings` меняется синхронно с листом и попадает в тот же автосейв |

Инварианты (проверяются в dev-сборке после каждой команды): блоки не пересекаются;
блок не вложен в блок; длина `rows` == высоте области данных; `anchor` в пределах листа.
Нарушение — лог + мягкая самопочинка (пересборка `rows` по маркерам) вместо падения.

### 1.5. Служебная разметка ячеек

Движку нужны только значения и стили; принадлежность ячейки блоку мы определяем
геометрией (`anchor` + размеры), а не пометкой каждой ячейки — снапшот не раздувается.
Единственная пометка на ячейке — кастомное свойство `ov:1` для override (движок Univer
позволяет custom metadata у ячейки) — чтобы рисовать уголок дёшево при рендере.

## 2. Обновление блока: алгоритм сверки

Вход: блок B, свежий результат его `query` (упорядоченный список строк с `entityKey`
и значениями по колонкам). Всё выполняется как **одна команда** движка (один шаг undo).

```
0. Автоверсия документа «до обновления блока …» (§I.3).
1. Снять текущее: rows[], overrides, значения free-строк.
2. Дифф по entityKey:
   added   = в свежем есть, в rows нет
   removed = в rows есть (не free/group), в свежем нет
   kept    = пересечение → сравнить значения по колонкам kind:"field"
3. Построить НОВЫЙ порядок строк:
   — свежий результат задаёт порядок строк данных и групп (sort/groupBy из query);
   — free-строки сохраняют относительную привязку: free-строка помнит entityKey
     ПРЕДЫДУЩЕЙ строки данных ("after": "tag:6f2c") и встаёт после неё в новом
     порядке; если та удалена — после ближайшей выжившей выше; сироты — в конец
     блока перед итогами (не теряем никогда).
4. removed-строки НЕ удалять молча: пометить missing:true, ячейки — стилем
   «серый + зачёркнутый». Панель блока: «3 строки выпали из выборки —
   [Удалить все] [Оставить как текст]». (Оставить = строки становятся free.)
5. kept: для каждой ячейки kind:"field":
   — нет override → записать новое значение;
   — есть override и живое значение НЕ изменилось с прошлого обновления → оставить;
   — есть override и живое значение изменилось → конфликт (§2.6), значение не трогаем.
6. added: вставить строки в свои позиции, залить значения, формульные колонки —
   развернуть шаблон, применить стиль блока (banding и т.п.).
7. rownum-колонки перенумеровать; итоги и группы пересобрать.
8. Поправить внешние формулы: движок сам сдвигает ссылки при вставке/удалении строк
   (команды вставки штатные); отдельно расширяем диапазоны вида «вся колонка блока»
   (=СУММ по D2:D40 → D2:D47), т.к. вставка В КОНЕЦ диапазона штатно его не растит.
   Правило: если диапазон формулы совпадал с колонкой области данных блока по
   границам — растянуть до новых границ. Только точное совпадение, никакой магии.
9. Обновить state.lastRefreshAt, снять stale, записать в SystemChangeLog.
```

Отмена обновления = обычный undo (вся сверка — одна команда) или откат на автоверсию.

### 2.1. «Обновить все данные»

Кнопка на тулбаре: одна автоверсия на документ, затем блоки последовательно (порядок —
по листам сверху вниз), один сводный отчёт: «Обновлено 4 блока: +12 строк, 37 изменений,
2 конфликта, 3 выпавших». Отчёт кликабелен — прыжки к местам изменений.

### 2.2. Что считается «изменением» значения

Сравнение — по нормализованному значению (§5.3: число как число, строка после trim),
а не по строковому представлению: смена «1250» ↔ «1 250,0» изменением не считается.

### 2.3. Изменение состава колонок проекта

Если параметр колонки исчез из проекта вовсе (каталог его больше не знает): колонка
остаётся, ячейки kept-строк не трогаем, у новых строк — пусто; на заголовке — значок
«поле не найдено в проекте» с подсказкой открыть алиасы. Ничего не удаляем сами.

### 2.4. Индикатор устаревания (stale) — как считается дёшево

`POST /query/check-fresh`: на сервер едет список `{blockId, entity, since}`. Сервер
держит по проекту счётчики «последнее изменение по типу сущности» (обновляются в тех же
местах, где шлются socket-события). Ответ — по типу сущности, без исполнения фильтров:
допускаем ложноположительный stale (изменили тег, не входящий в выборку) — это честная
цена за мгновенный ответ; ложноотрицательных нет.

### 2.5. Overrides

Ключ — `r:<entityKey>|c:<colId>` (не координаты! строки ездят). Хранится значение
пользователя, автор, время и `baseValue` — живое значение на момент правки (для
детекции конфликта в §2.5 шаг 5). Снятие override: контекстное меню ячейки
«Вернуть значение из проекта» и массово в панели блока.

### 2.6. Конфликты

```jsonc
{ "rowKey": "tag:6f2c", "colId": "col_h", "userValue": "1300",
  "liveValue": "1350", "baseValue": "1250", "detectedAt": "…" }
```

UI — как разрешение конфликтов ревизий в «Оборудовании» (знакомый паттерн): панель со
списком, по строке: [Моё] [Из проекта], сверху [Все мои] [Все из проекта]. Пока конфликт
не решён, ячейка с оранжевым уголком; печать/экспорт не блокируем (печатается текущее
значение), но в диалоге экспорта — предупреждение «в документе N нерешённых конфликтов».

## 3. Грамматика путей полей

Единый адрес поля — строка, используется в колонках, фильтрах, каталоге и функциях:

```
path        := simpleField | paramField | dictField | relField | metaField | serviceField
simpleField := ident                      // поле своей сущности: "identifier", "brand"
relField    := rel "." path               // переход по связи: "equipment.system.name",
                                          // "tag.department" (от элемента к тегу)
paramField  := "param:" group "|" key     // параметр элемента: "param:Габариты|Высота"
metaField   := "meta:" jsonPath           // из Tag.metadata: "meta:procurement.stage"
dictField   := "dict:" dictName "." attr  // разыменование кода через справочник:
                                          // "dict:Отделения.nameRu" применительно к полю-коду
serviceField:= "#" ident                  // "#rownum", "#today", "#user", "#project.customer"
```

Правила разрешения:

1. **Связи** (`rel`) допускаются по фактическим FK: tag↔element (m:n — берём первый
   связанный, при >1 значение «эл1; эл2» + пометка), element→monoblock→system,
   tag→project, file→tags. Максимум 3 перехода — дальше каталог просто не предлагает.
2. **`param:`** сначала прогоняется через таблицу алиасов проекта (§5.4): алиас — это
   именованный набор пар «группа|ключ»; путь может ссылаться и на алиас:
   `param:@Расход` (символ `@` = имя алиаса).
3. **`meta:`** — точечный jsonPath без выражений (только ключи и индексы). Закупки —
   это `meta:procurement.stage`, лог этапов — `meta:procurement.stageLog.<код>`.
   Каталог строит список реально встречающихся meta-путей так же, как параметров.
4. Поле не разрешилось (нет параметра, нет связи) → в данных пусто, в UI на заголовке
   значок; функции формул в этом случае дают типизированную ошибку (§6.3). Никогда
   не «падаем» — путь может стать валидным после следующего импорта.

Пути — единственный контракт между клиентом и сервером о полях. Парсер и резолвер —
общий модуль (используется сервером в query и клиентом в подсказках), с юнит-тестами
на каждую ветку грамматики.

## 4. Исполнитель запросов: серверная механика

### 4.1. Материализация вместо SQL-эквилибристики

Динамические поля лежат в JSON-колонках (`specs`, `metadata`) — фильтровать их в SQLite
через json_each можно, но код будет хрупким, а планы — непредсказуемыми. Вместо этого:

- **проектный срез** (project snapshot) — плоские массивы записей по каждой сущности с
  уже распарсенными specs (+ наложенными `overrides` элемента — общий модуль с экраном
  «Оборудование», §I.4.3) и metadata; строится одним проходом по Prisma-запросам с
  `include`, кэшируется в памяти процесса на `projectId`;
- инвалидация — теми же событиями, что и каталог (изменение тега/элемента/импорт);
  кэш и каталог — один объект, каталог = агрегаты над срезом;
- фильтры/сортировка/группировка исполняются по срезу в JS. Для наших объёмов
  (тысячи тегов, десятки тысяч параметров) это микросекунды-миллисекунды; измеряем в
  бенчмарке (§13), порог пересмотра решения — 100 мс на типовой запрос.

Память: срез проекта ~ единицы МБ; держим LRU на 3 последних проекта.

### 4.2. Операторы фильтров

`eq, neq, contains, ncontains, startsWith, in, nin, empty, nempty, gt, gte, lt, lte,
between, dateBefore, dateAfter, dateBetween`. Числовые сравнения — по нормализованному
числу (§5.3); если значение не парсится числом — строка сравнением локали ru.
Группировка условий: одна степень вложенности `{all:[...]} / {any:[...]}` —
покрывает практически всё и не требует конструктора скобок в UI.

**Динамический операнд из ячейки**: `{"op":"eq","valueFrom":{"cell":"Настройки!B1"}}` —
значение снимается с листа в момент обновления блока. Так документ становится
параметризуемым без пересборки: сменил в жёлтой ячейке «У1» на «У2» → «Обновить» →
таблица по другой системе. В мастере это галочка «брать значение из ячейки…».

### 4.3. Агрегаты и итоги

Сервер отдаёт только строки; итоги (`СУММ`, `СРЗНАЧ`, `СЧЁТ`, `МИН`, `МАКС`) блок пишет
**формулами листа** по колонке — прозрачно для пользователя (видно и можно поправить),
и не требует серверного языка выражений. Подзаголовки групп — строки `{group}` в
реестре; итог группы — формула по диапазону группы, пересобирается при сверке (§2 шаг 7).

### 4.4. Ответ query

```jsonc
{ "rows": [{ "key": "tag:6f2c", "cells": ["3700-K02", "Systemair", 1250, …],
             "route": "/registry?tag=6f2c…" }],   // для клик-перехода
  "total": 247, "truncated": false,
  "columnsMeta": [{ "path": "…", "type": "number", "unit": "мм", "fillRate": 0.79 }] }
```

`route` — тот же механизм сквозных переходов, что в `SystemChangeLog.targetRoute`.
Предпросмотр мастера — тот же endpoint с `limit: 20`.

## 5. Каталог полей: построение и нормализация

### 5.1. Построение

Один проход по срезу (§4.1). Для каждого узла-поля копим: количество носителей,
`fillRate`, инференс типа, единицу, до 20 самых частых значений (для автоподстановки в
фильтрах и оператора `in` с чекбоксами), min/max для чисел и дат.

### 5.2. Инференс типа значения параметра

Значения specs — строки из импортированных бланков. Классификатор по образцу значения:

```
"1250"        → number
"1 250,5"     → number (пробелы-разряды, запятая-десятичная)
"1250 мм"     → number + unit "мм" (unit также приходит отдельным полем specs — он главнее)
"230/400 В"   → string (не терзаем: составные значения не числа)
"12.05.2025"  → date (dd.mm.yyyy | yyyy-mm-dd)
"да/нет/—"    → string
```

Тип поля = тип ≥80% непустых значений, иначе `mixed` (фильтры строковые, сортировка
«числа как числа, потом строки»). Инференс — только подсказка UI и сортировки;
исходная строка никогда не искажается при показе.

### 5.3. Нормализация числа

Общая функция `parseRuNumber` (сервер+клиент, один исходник в `src/lib` /
переиспользуемый сервером): убрать пробелы/неразрывные пробелы, запятую → точку,
отрезать хвост-единицу. Используется: фильтры, сортировка, сравнение при сверке (§2.2),
типизация ячеек при вставке блока (числа кладём числами — иначе у пользователя не
заработает `=СУММ`).

### 5.4. Алиасы параметров

Хранение: `AppSetting`, ключ `constructor_param_aliases`, userId=null (общие на проект,
т.к. шаблоны общие); значение:

```jsonc
{ "проект-id": [ { "name": "Расход", "unit": "м³/ч",
                   "members": ["Вентилятор|Расход воздуха", "Вентилятор|Расход, м3/ч",
                               "Общие|Производительность"] } ] }
```

Правила: пара «группа|ключ» может состоять максимум в одном алиасе (валидируем);
разрешение значения по алиасу — первый непустой member в порядке списка. UI — экран в
настройках Конструктора: слева дерево параметров с частотами, справа алиасы;
«похожие» кандидаты подсвечиваются (нормализованный Левенштейн по ключам ≥0.7 —
переиспользуем подход из дизайна ИИ-чата). Мастер (§I.5, шаг 3) показывает алиасы
поверх сырых параметров и вежливо предлагает создать алиас, когда пользователь выбирает
параметр с fillRate < 50%, у которого есть «похожий» сосед.

### 5.5. Скрытые параметры

`equip_visibility` (токены `g:`/`p:`) — это **настройка отображения карточки**, не
данных: каталог показывает всё, но скрытые — приглушённо с пометкой «скрыт в карточке
оборудования». Логика «раз скрыл там — не предлагать тут» была бы сюрпризом: для
заказчика часто нужны как раз поля, скрытые в рабочей карточке.

## 6. Функции формул: механика

### 6.1. Регистрация и имена

Кастомные функции регистрируются в движке пакетом «Flux». Каждая — RU-имя + EN-синоним
(`ТЕГ/TAG`, `ПАРАМ/PARAM`, `ПРОЕКТ/PROJECT`, `СПРАВ/DICT`, `СЧЁТ_ТЕГОВ/COUNTTAGS`,
`ПОЛЬЗОВАТЕЛЬ/USERNAME`). Разделитель аргументов — `;` (ru-локаль движка), `,`
принимается парсером как запасной. Автодополнение и карточка-подсказка сигнатуры —
штатный механизм движка, описания даём по-русски.

### 6.2. Синхронный движок поверх асинхронных данных

Движок вычисляет синхронно, данные — на сервере. Схема:

```
1. При открытии документа: скан всех формул → сбор требуемых срезов
   (какие теги, какие параметры, нужен ли проект/справочники) → ОДИН batch-запрос
   `POST /api/constructor/fn-data` → клиентский кэш (Map по нормализованному ключу).
2. Вычисление функции: значение в кэше? → вернуть. Нет? → вернуть ошибку #ЗАГРУЗКА,
   положить ключ в очередь дозагрузки (дебаунс 150 мс, batch), по приходу данных —
   точечный пересчёт затронутых ячеек (API движка это умеет).
3. Кнопка «Обновить данные» и события stale инвалидируют кэш → пересчёт.
```

Кэш формульных данных и данные блоков — **разные** механизмы намеренно: блоки
обновляются только явно (§2), формулы — «живее» (при открытии всегда свежие). Это
интуитивно совпадает с ожиданием: формула = «текущее значение», блок = «собранный отчёт».

### 6.3. Ошибки

| Код | Когда | Поведение |
|---|---|---|
| `#ЗАГРУЗКА` | данных ещё нет в кэше | исчезает сам после дозагрузки |
| `#НЕТ_ТЕГА` | идентификатор не найден | подсказка в тултипе: «проверьте идентификатор» + похожие |
| `#НЕТ_ПАРАМА` | путь не разрешился (§3.4) | тултип: группа/ключ + ссылка «открыть алиасы» |
| `#МНОГО` | `=ПАРАМ` по тегу с >1 элементом и неоднозначным параметром | тултип: «уточните itemCode: …» |
| `#ДОСТУП` | нет права на срез | по правам §11 |

Ошибки — типизированные значения движка (как штатный `#N/A`), участвуют в ЕОШИБКА/IFERROR.

### 6.4. Экономия при следующем открытии

После скана формул список требуемых срезов кладём в `bindings.cellFunctions` — при
следующем открытии batch-запрос уходит немедленно, не дожидаясь ленивого скана.
`СЕГОДНЯ()`/`ПОЛЬЗОВАТЕЛЬ()` — volatile движка, серверу не нужны.

## 7. Хранение, автосейв, целостность

### 7.1. Формат и размер

Снапшот книги — JSON движка. Оценка: ячейка со стилем ~120–200 байт JSON; документ
50 000 ячеек ≈ 6–10 МБ текста. Решения:

- в SQLite пишем **gzip-блоб** (колонка `workbook` типа Bytes в отдельной таблице или
  base64 в String — на спайке замерить; ожидание: сжатие 8–15×, т.е. типовой документ
  < 1 МБ);
- версии (§I.3) храним тоже сжатыми; политика удержания: последние 10 подряд + по одной
  на день за последние 30 дней, остальное схлопывается фоновой задачей;
- лимит-предохранитель: при снапшоте > 25 МБ сырого JSON — предупреждение «документ
  очень большой, вынесите данные на отдельные листы/документы», при > 50 МБ — запрет
  автосейва новых гигантских вставок (вставку блока режем лимитом §I.12 раньше).

### 7.2. Автосейв

Дебаунс 2 с + принудительно каждые 30 с при непрерывном редактировании + при уходе с
экрана/закрытии окна (`beforeunload` → синхронный best-effort). PUT несёт
`baseUpdatedAt`; сервер сравнивает с текущим `updatedAt` документа:

- совпало → пишем;
- не совпало (кто-то сохранил позже — возможно при перехвате блокировки, §10) → `409` +
  свежая мета; клиент показывает: «Документ изменён (Иванов, 14:02). [Взять его версию]
  [Сохранить мою как копию]». Тихо не перетираем ничьё.

### 7.3. Восстановление после сбоя

Несохранённые изменения дублируются в localStorage (`constructor_pending_<docId>`:
последний неотправленный снапшот, сжатый). При открытии документа: если pending новее
серверного `updatedAt` — предложить восстановить. После успешного PUT — чистим.
Это закрывает «свет мигнул» на portable-версии.

### 7.4. Битый снапшот

Открытие в try/catch + валидация схемы. Не открылся → **безопасный режим**: документ
открывается из последней версии, баннер «текущее состояние повреждено, открыта версия
от …». Повреждённый блоб не удаляем (кладём в `versions` с пометкой corrupt) — для
разбора. Редактор обёрнут в React error boundary: падение движка не роняет приложение,
предлагает перезагрузить раздел.

## 8. Печать и PDF: пагинация

### 8.1. Пайплайн

```
снапшот → печатная модель (общая с экспортом, §9) → HTML (инлайн-стили, @page)
→ скрытое BrowserWindow → measure (реальные высоты строк) → расстановка разрывов
→ webContents.printToPDF / print
```

Меряем реальным рендером, а не расчётом метрик шрифта — иначе переносы строк в ячейках
(«перенос текста» включён) не угадать.

### 8.2. Алгоритм разрывов

1. Область печати листа: заданная пользователем или автообласть (по непустым ячейкам).
2. Ширина: колонки, попавшие в область; режимы «масштаб N%» или «вписать по ширине»
   (масштаб = ширина_страницы / суммарная_ширина_колонок, ограничен снизу 40% с
   предупреждением «таблица слишком широкая — включите альбомную»). Разрыв ПО ВЕРТИКАЛИ
   колонок в v1 не делаем (широченные таблицы → альбомная/масштаб/меньше колонок);
   помечено как «потом» — Excel-овское «страница 2 по горизонтали» нужно редко.
3. Высота: жадная укладка строк; строка не рвётся посередине (строка выше страницы —
   единственное исключение, режем с маркером «…продолжение»).
4. **Повтор шапки**: строки шапки блока (`headerRows`) и/или явно заданные пользователем
   «сквозные строки» повторяются на каждой странице.
5. Объединённая ячейка на границе страницы: разрыв переносится ВЫШЕ начала объединения,
   если это не делает страницу пустее 30% — иначе объединение рвётся с повтором
   содержимого и пометкой.
6. «Висячие» строки: подзаголовок группы не остаётся последней строкой страницы
   (переносим на следующую вместе с ≥1 строкой данных).
7. Колонтитулы: шаблоны с переменными `{документ} {проект} {заказчик} {дата}
   {страница} {всего}`; «N из M» — двухпроходно (первый проход считает M).
8. Ч/б-режим печати: заливки → нет, цветные шрифты → чёрный, banding → тонкие линии.
```

### 8.3. Шрифты

В печатной модели — только гарантированные на Windows семейства (список в UI выбора
шрифта редактора сознательно короткий: Calibri, Arial, Times New Roman, Courier New +
системный). Экзотика из импортированного .xlsx маппится на ближайший с пометкой.
printToPDF вшивает системные шрифты сам — отдельной возни с embedding нет.

## 9. Матрица соответствия экспорта

Одна **печатная/экспортная модель** (лист → строки → ячейки: значение, тип, формат
числа, шрифт, заливка, границы, объединения, ширины) → три бэкенда. Что куда переносится:

| Возможность | XLSX (exceljs) | PDF/печать | DOCX |
|---|---|---|---|
| значения, типы, форматы чисел | ✔ | ✔ (отрендеренный текст) | ✔ (текст) |
| формулы | ✔ **режим на выбор**: формулы или значения¹ | значения | значения |
| шрифты/цвета/заливки/границы | ✔ | ✔ | ✔ |
| объединения | ✔ | ✔ (§8.2.5) | ✔ |
| ширины/высоты | ✔ (px→chars: ≈px/7, px→pt: ×0.75) | ✔ | приблизительно² |
| фильтры (автофильтр) | ✔ | — | — |
| закрепление областей | ✔ | — | — |
| условное форматирование | ✔ как **вычисленные статические стили**³ | ✔ (то же) | ✔ (то же) |
| несколько листов | ✔ | ✔ (последовательно, каждый со своей ориентацией) | ✔ (разделы) |
| изображения | ✔ | ✔ | ✔ |
| комментарии ячеек | ✔ | — (опция «списком в конце») | — |
| внутренние ссылки на теги | текстом⁴ | текстом | текстом |
| умные блоки | обычные ячейки (привязки — наша фишка, наружу не едут) | — | — |

¹ Формулы наших функций (`=ТЕГ()`) в Excel не существуют — при экспорте «с формулами»
они всегда заменяются значениями, остальные (`СУММ`, `ЕСЛИ`…) едут формулами.
² Word-таблица живёт своей жизнью: экспортируем проценты ширины от печатной области.
³ Честно и предсказуемо: получатель видит ровно то, что видел автор; правила CF
переносить exceljs умеет частично — не связываемся в v1.
⁴ `flux://…`-ссылки вне программы бессмысленны.

Матрица — прямо в UI: диалог экспорта показывает короткое «что не переносится» под
выбранным форматом. Никаких сюрпризов у получателя.

### 9.1. Безопасность содержимого

- **Вставка из буфера/импорт .xlsx**: значения, начинающиеся с `=`, вставляются как
  текст, если источник — не наш документ (иначе чужой файл может привнести формулу);
  включить как формулу — явное действие («Вставить как формулы»).
- **Экспорт**: текстовые значения, начинающиеся с `= + - @`, в exceljs пишем строковым
  типом ячейки (никогда формулой) — классическая формула-инъекция не проходит.
- Пути экспорта в Проводник — через существующий механизм FileNode (валидация имён там).

## 10. Совместная работа (важно для Postgres-режима)

> **Пересмотрено частью IV** (`constructor-design-v0.25-collab.md`): в сетевом
> (Postgres) режиме вместо блокировки — полноценное совместное редактирование
> (OT-lite поверх op-лога). Протокол ниже остаётся для LOCAL-режима и как фоллбэк.

Приложение умеет общую PostgreSQL — несколько человек работают одновременно, поэтому
протокол блокировки не «на вырост», а обязательный:

```
Открытие редактора → POST /docs/:id/lock {userId}
  свободен/протух → lock: {userId, name, heartbeatAt}; ответ acquired
  занят           → ответ {lockedBy}; клиент открывает READ-ONLY с кнопкой «Перехватить»
Heartbeat каждые 15 с (PUT lock); TTL 45 с — три пропуска = протух.
Перехват → lock переписывается; старому владельцу socket-событие
  constructor:lock_lost → его редактор мгновенно замирает в read-only с баннером
  «Документ перехвачен: Иванов»; его несохранённое уходит в localStorage-журнал (§7.3)
  и предлагается «сохранить как копию».
Закрытие/уход с экрана → DELETE lock (best-effort; TTL подчистит, если не дошло).
```

Read-only-режим — полнофункциональный просмотр: скролл, выделение, копирование,
экспорт/печать (право `constructor.export`), просмотр версий. Live-подглядывание чужих
правок в v1 не делаем (см. отвергнутое, §16.4) — read-only показывает состояние
последнего автосейва, кнопка «Обновить вид».

Socket-события раздела: `constructor:doc_created|updated|deleted` (обновление
Библиотеки), `constructor:lock_lost`, `constructor:stale` (проектные данные изменились —
открытые документы включают значок stale без поллинга).

## 12. Версии: дифф и UX

- Список версий: время, автор, комментарий, размер, происхождение (авто/ручная/
  перед обновлением/перед восстановлением).
- **Просмотр версии** — открытие снапшота в read-only-редакторе (никакого отдельного
  вьюера — тот же компонент).
- **Дифф версий**: по ячейкам сравниваем значения (не стили): списком
  «Лист!B12: 1250 → 1350», кликабельно, с фильтром по листу; при > 2000 отличий —
  честное «изменений слишком много, откройте версии рядом». Стили не диффаем (шум).
- Восстановление: текущее состояние сначала само становится версией («перед
  восстановлением…») — откат отката всегда возможен.

## 14. Тест-план

1. **Юнит**: парсер путей (§3) — вся грамматика + мусорные входы; `parseRuNumber` и
   инференс типов (§5.2–5.3) — таблица образцов; операторы фильтров; алгоритм сверки
   (§2) — изолированно от движка (вход: rows+свежие данные, выход: список операций) —
   именно здесь живут все крайние случаи: free-строки-сироты, missing, конфликты,
   группа опустела, всё удалено, всё новое.
2. **Golden-файлы экспорта**: фикстурный проект (мини-БД в репозитории) + 6 эталонных
   документов → XLSX/PDF/DOCX; сравнение XLSX — по распакованному XML (стабильные поля),
   PDF — по извлечённому тексту + количеству страниц. Ловит регрессии пагинации и стилей.
3. **Совместимость снапшотов**: каталог фикстур-снапшотов каждой прошлой версии
   формата; тест «все открываются» гоняется при любом обновлении движка — главный
   страховочный трос апгрейдов Univer.
4. **Протокол блокировок**: сценарии перехвата/протухания/409 — интеграционно на
   тестовом сервере.
5. **Бенчмарки** §13 — отдельный npm-script, прогон на спайке и перед релизами.
6. **Приёмка** — 10 сценариев §I.13 руками по чек-листу.

## 15. Крайние случаи — реестр (собран по всем разделам, для приёмки)

1. Тег связан с несколькими элементами → `param:` через тег: значение «эл1; эл2» в
   блоке, `#МНОГО` в функции без itemCode.
2. Два тега с одинаковым identifier (в разных системах) → функции по identifier дают
   первый + предупреждение в тултипе; блоки работают по id — им всё равно.
3. Параметр с одинаковым ключом в двух группах одного элемента → путь включает группу,
   коллизии нет; алиас может объединить сознательно.
4. Пользователь удалил колонку листа внутри блока → колонка блока помечается «оторвана»,
   при обновлении не заполняется; починка — «Изменить выборку».
5. Обновление при открытой панели конфликтов → панель пересобирается, решённые
   конфликты не воскресают (ключ конфликта включает `liveValue`).
6. Документ открыт, проект переключили в шапке приложения → редактор остаётся на своём
   `projectId` (документ принадлежит проекту), баннер «документ из проекта X».
7. Шаблон применён к проекту без единого совпадения полей → документ создаётся со
   структурой и пустыми блоками + мастер сопоставления сразу открыт.
8. Импортированный .xlsx с 200 000 ячеек → предупреждение + предложение отрезать
   пустые области; жёсткий предел из §7.1.
9. Обновление блока, где пользователь отсортировал строки вручную через блок-панель
   (изменился `query.sort`) → порядок применяется свежий, free-строки по правилу «after».
10. Печать документа с нерешёнными конфликтами/`#ЗАГРУЗКА` → предупреждение в диалоге;
    `#ЗАГРУЗКА` блокирует экспорт до дозагрузки (секунды), конфликты — нет.
11. Часы на машине переведены назад → версии сортируем по (createdAt, id), TTL блокировок
    считаем по серверному времени только.
12. В `metadata` тега лежит невалидный JSON → срез помечает тег «metadata нечитаемо»,
    каталог не падает, поле пустое.

## 16. Инвентарь взаимодействий (систематически, вместо «1000 вариаций»)

Пространство возможностей — декартово произведение, и оно всё покрывается уже принятыми
механизмами. Это — проверка полноты дизайна, а не список работ:

### 16.1. Оси

- **Источник**: теги | элементы | системы | моноблоки | файлы | закупки (meta) |
  справочники | проект | пользователи (для листов согласования).
- **Строки**: всё | фильтры (16 операторов × И/ИЛИ × значение-из-ячейки) | ручной выбор.
- **Колонки**: поле | параметр | алиас | связь (до 3 переходов) | meta-путь |
  формульная | №п/п | статическая.
- **Форма**: строки | транспонированно | с группами | с итогами | с группами и итогами.
- **Жизнь данных**: статичная вставка | живой блок | одиночная функция.
- **Выход**: экран | печать | PDF | XLSX (формулы/значения) | DOCX | Проводник | диск.

Любая комбинация осей должна работать без специального кода под неё — если на приёмке
находится комбинация, требующая костыля, это дефект архитектуры, а не «ещё фича».

### 16.2. Микровзаимодействия редактора (обязательные к v1)

Статус-бар с агрегатами выделения (сумма/среднее/кол-во — самое используемое в Excel);
маркер автозаполнения; двойной клик по границе колонки — автоширина; Ctrl+стрелки —
прыжки по данным; Ctrl+Shift+L — фильтр; F2 — редактирование; Esc — отмена ввода;
контекст строки блока → «Открыть тег/элемент», «Сделать свободной», «Вернуть значения
из проекта»; перетаскивание поля из панели «Данные проекта» на заголовок колонки
блока → добавить колонку в блок; на пустую ячейку → вставить `=ПАРАМ(...)`; зум
Ctrl+колесо; тёмная тема (движок темизируется, печать — всегда светлая).

### 16.3. Точки входа в Конструктор из других разделов

- Реестр тегов: выделил строки → «Отправить в Конструктор» (мастер с предвыбором).
- Карточка оборудования: «Собрать таблицу по таким же» (фильтр по equipType).
- Менеджмент: текущий отфильтрованный вид → блок закупок.
- ИИ-ассистент: намерение «собери таблицу …» → мастер с предзаполненными шагами
  (каталог полей + стеммер из дизайна ИИ-чата дают разбор «высота, ширина, вес по
  всем приточкам»).

### 16.4. Отвергнутое — и почему (чтобы не возвращаться случайно)

| Идея | Почему нет |
|---|---|
| Автообновление блоков в реальном времени | перезапись под руками разрушает доверие; stale-значок + кнопка дают то же без риска |
| CRDT-совместное редактирование | CRDT не нужен (сервер один — офлайн-ветвлений истории нет); совместное редактирование сделано проще — OT-lite с серверной сериализацией, см. часть IV |
| Серверный язык выражений в query | дублирует формулы листа; вычисляемые колонки (§1.2) решают то же прозрачнее |
| Макросы/скрипты | безопасность + поддержка; функции и блоки закрывают потребность |
| Ссылки между документами Конструктора | клубок зависимостей и загадочные пересчёты; экспорт листа в другой документ — копией |
| Правка данных ПРОЕКТА из ячеек блока (обратная запись) | опасно и неочевидно (какая ячейка «главная»?); переход к карточке — правильный путь правки |

### 16.5. Кандидаты «потом» (осознанно за рамками фаз §I.15)

Диаграммы по диапазону; сводные (если Univer-платные не понадобятся — своя лёгкая
сводка мастером «группировка+агрегаты» уже близка); горизонтальная пагинация печати;
экспорт CSV (просто, но ждём запроса); подпись/штамп на PDF; расписание автоэкспорта
(«каждый понедельник — PDF в папку заказчика») — хорошо ляжет на существующие механизмы,
но пусть попросят.

## 17. Открытые вопросы к спайку (Фаза 0)

1. Скорость батч-вставки 50 000 ячеек в Univer одной командой; если медленно —
   вставка снапшот-мутацией с полным перерендером листа.
2. Достаточно ли command-интерцепторов Univer для всех строк таблицы §1.4
   (особенно сортировка диапазона и вырезание/вставка) — иначе часть действий
   запрещаем внутри блока с подсказкой «используйте панель блока».
3. Реальные размеры снапшота и степень сжатия на типовом документе (§7.1).
4. Темизация под тёмную тему без вмешательства в исходники движка.
5. Кастомное свойство ячейки для маркера override (§1.5) — переживает ли undo/copy.
6. Качество чтения .xlsx: связка «xlsx для данных + перенос базовых стилей» против
   импорта только значений (и честного предупреждения).

Каждый пункт — с готовым планом Б, ни один не блокирует архитектуру: слой привязок,
каталог, query, экспорт и печать от ответа не зависят (движок изолирован адаптером —
`bindingSync` и вставка/чтение снапшота — единственные точки контакта).
