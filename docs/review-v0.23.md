# Ревизия программы v0.23 (только анализ, без применения)

Просмотр кода на предмет: быстродействия, сжатых/сломанных разделов, тёмной темы,
и улучшений ИИ-чата. Ничего не изменено — это карта того, что стоит поправить.

---

## 1. Быстродействие — что тормозит и почему

### 1.1 Рабочий чат опрашивает сервер каждые 3 секунды
`chatStore.ts:569` — `setInterval(fetchMessages, 3000)`. При этом параллельно поднимается
socket.io (`setupSocket`). То есть в проде чат и слушает сокет, и долбит сервер каждые 3 с.
Смягчено `setMessagesIfChanged` (не перерисовывает, если список не изменился), но сам
запрос + разбор JSON идёт всё равно.
→ Когда сокет подключён — опрос сделать резервным (раз в 20–30 с), а не основным.

### 1.2 Уведомления опрашиваются каждые 15 с всегда
`notificationStore.ts:77` — поллинг не выключается, даже когда окно свёрнуто/не в фокусе.
В логе видно `/notifications` каждые 15 с всю сессию.
→ Тормозить опрос при `document.hidden`, возобновлять при фокусе.

### 1.3 Registry — монолит на 5491 строку и 74 `useState`
Один компонент обслуживает 4 вкладки (граф/дерево/экспорт/спецификация). Вкладки
рендерятся условно (это хорошо), но **74 разрозненных стейта в одном компоненте**:
любое изменение (наведение, символ в поле, выбор карточки) заново прогоняет весь
render-функционал на 5000 строк. Часть уже вынесена в мемо (`TagSearchPanel`), но этого мало.
→ Разбить на `BoardTab / TreeTab / SegmentsTab / SpecTab` — каждый со своим стейтом;
верхний компонент держит только общие данные. Резко срежет стоимость перерисовок.

### 1.4 Нет общего кэша данных проекта
`Registry`, `Equipment`, `Explorer`, `ProcurementManagement` — каждый при заходе сам
тянет `/tags`, `/systems`, `/dictionaries`. Переключение разделов = повторная загрузка
тех же данных (в логе видно по 3–4 одинаковых запроса за клик).
→ Общий стор с TTL-кэшем (например 10–15 с) на теги/системы/словари; экраны читают из него.

### 1.5 Прочее
- `messagesSignature` строит длинную строку по всем сообщениям каждые 3 с — на больших
  переписках заметно; можно сравнивать по `updatedAt`+кол-ву.
- Массовые операции в некоторых местах ещё идут циклом `await` (демо-данные Registry).
  Уже есть `PUT /api/tags/bulk-metadata` — переиспользовать везде.

## 2. Тёмная тема — где ломается отображение

### 2.1 Модалки «Проводника» — жёстко белые (подтверждённый баг)
`Explorer.tsx`: всплывающее меню создания (1414), окно переименования (1439),
окно привязки тегов (1554), кнопки сортировки (1568–1574), кнопка «Отмена» (1627) —
используют `bg-white`, `text-slate-800`, `bg-slate-200` **без `dark:`-вариантов**.
В тёмной теме это белые окна с тёмным текстом на белом — выглядит сломанно.
→ Добавить `dark:bg-slate-900 / dark:text-slate-100 / dark:border-slate-800` и т.п.

### 2.2 Печать/экспорт заметок — светлые инлайновые стили
`NotesManagement.tsx:237,262` — `color:#0f172a` на белом фоне. Это **для печатного
листа Word/принтера, так и надо** — не баг, отмечаю чтобы не «чинить» зря.

### 2.3 Общий прогон
Остальные экраны (Login, Sticker, Dashboard) тёмную тему покрывают. Точечно стоит
пройти все модалки и поповеры — исторически именно они забывают `dark:`.

## 3. Сжатые / неудобно расположенные разделы

### 3.1 Карточка оборудования — пустая трата ширины
`Equipment.tsx:444` — параметры выводятся одним столбцом «подпись … значение».
На широком экране (как на скриншоте с клапаном РЕГУЛЯР-Л) справа огромные пустые поля,
а список тянется вниз на десятки строк.
→ Адаптивная сетка в 2–3 колонки карточек-параметров: и компактнее, и вся ширина в дело.

### 3.2 Экспорт/импорт и Спецификация в Registry
Ранее отмечены в плане переделок (разделы 4–5): простыни фильтров по сегментам с
подзаголовками, много воздуха, длинная прокрутка до кнопки. Остаются кандидатами на
компактный трёхзонный макет + живой предпросмотр (детали — в `redesign-plan-v0.23.md`).

### 3.3 Заметки — фиксированная высота
`NotesManagement.tsx:345` — `h-[calc(100vh-100px)]`. На невысоком окне редактор
подрезается. → Перейти на `flex-1 min-h-0`, чтобы тянулось по факту.

## 4. ИИ-чат — что улучшить

Сейчас (`assistantStore.ts` + `knowledge.ts` + `tours.ts`): ключевые слова + локальная
база + подбор демонстраций. Работает офлайн — это плюс, оставляем. Слабые места:

### 4.1 Плохо переносит опечатки и склонения
`tokensFrom` + `fieldMatches` отрезают только 2 буквы. «спецификацыя», «менеджмнт»,
«вентиляторов» в косвенных формах теряются.
→ Лёгкий стеммер (основа слова) + допуск 1 опечатки (Левенштейн) без внешних библиотек.

### 4.2 Узкий охват данных
Отвечает только про теги и компоненты. Не умеет: этап закупки позиции (Менеджмент),
дубликаты кодов, статус проекта, содержимое заметок, «кто что менял» (логи).
→ Расширить домен: «на каком этапе 3700-K02», «покажи дубли», «что в работе»,
«последние изменения».

### 4.3 Нет контекста диалога
«А сколько их?» после «покажи теги вентиляции» не понимается.
→ Хранить последний предмет разговора и подставлять.

### 4.4 Ответы без действий
Находит теги — выдаёт таблицу. Нет кнопок «Открыть на холсте», «Показать в Менеджменте»,
«Найти дубли», «Создать заметку», «Открой раздел…».
→ Добавить action-кнопки и навигационные команды.

### 4.5 Мелочи
- Порог `demoMode` 1.5 иногда даёт нерелевантную демонстрацию — калибровать.
- База знаний статична; проверить, что названия разделов актуальны (после редизайнов).
- Честное «не знаю» с 3 подсказками тем вместо общего фоллбэка.

## 5. Разное (мелкие недочёты)

- `server.ts` содержит литеральные управляющие символы в регэкспе очистки (из строки
  `clean()`). Работает и собирается, но файл помечается как «binary» в grep — для
  поддерживаемости лучше перевести на `\uXXXX`-escape.
- Чат в проде: путь «сокет + опрос» может обработать одно сообщение дважды; сейчас
  спасает полная замена массива в `setMessagesIfChanged`, но при переходе на инкремент
  нужно дедуплицировать по `id`.
