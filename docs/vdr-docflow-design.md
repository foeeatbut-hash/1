## 1. ВДР — реестр документации (ядро документооборота)

### Что такое ВДР (из присланных файлов)
**Vendor Document Register** — сводная таблица всех документов поставщика по заказу.
Структура реального файла:
- **Titular sheet** — шапка: проект, подрядчик (Wison), заказчик (ZapSib), поставщик (ВЕЗА),
  № проекта, № заказа (PO `22062-PEQ-0371`), № документа (Contractor/Owner/Vendor).
- **Accounting for revisions** — учёт ревизий: `A,B,C…` (черновые, IFR) → `0,1,2,3…`
  (Certified Final), дата, описание.
- **Vendor Document Register** — строки-документы, колонки:
  `Vendor · Purchase Order · Contractor Doc No · Owner Doc No · English Title ·
  Russian Title · Doc.Type (VDR code) · Revision · Date · Reason for Issue · Language`.

Реальный масштаб: 562 документа, ~52 типа (VDR codes). Примеры типов:
`C01 General Arrangements (61), C32 3D Equipment (152), E02 Equipment Data Sheets (75),
K32 Technical Passport (91), D-серия схемы, F-серия списки, K-серия сертификаты, R-серия
реестры/руководства`. Причины выпуска: `IFR` (на рассмотрение), `IFI` (для сведения),
`CEF` (Certified Final). Ревизии: буквенные (черновые) → цифровые (утверждённые).

### Модель данных (новые таблицы)

```
DocRegister (ВДР-реестр — «исходник»)
  id, projectId, name ("ВДР 22062-PEQ-0371")
  vendor, contractor, owner, poNumber, ownerProjectNo, contractorProjectNo
  sourceDocId?  (ConstructorDoc.id — если ВДР ведётся таблицей в программе)
  createdAt, updatedAt

DocRegisterItem (строка реестра = карточка документа)
  id, registerId, projectId
  contractorNo, ownerNo, vendorNo         (номера документа)
  titleEn, titleRu
  vdrCode           (тип: C01, E02, K32…)  → справочник VdrType
  revision          (A/B/…/0/1/2)
  issueDate, reasonForIssue (IFR/IFI/CEF), language
  equipmentTags     (JSON: привязка к тегам/оборудованию — «какому оборудованию»)
  status            (D черновик → C на проверке → B согласован → A выдан заказчику — уже есть статусы)
  fileNodeId?       (ConstructorDoc/FileNode — сам сформированный документ)
  assigneeId?       (инженер-исполнитель), managerId? (ответственный менеджер)
  meta (JSON)

VdrType (справочник кодов ВДР — общий, редактируется в Справочнике)
  code (C01), titleEn, titleRu, discipline (HVAC/EL/INSTR…)
```

### Как ВДР попадает в программу (3 пути, «многообразие»)
1. **Импорт готового Excel-ВДР** (как присланные файлы): мастер разбирает лист
   «Vendor Document Register», сопоставляет колонки (авто по заголовкам EN/RU),
   титульник → реквизиты реестра. Создаёт `DocRegister` + `DocRegisterItem[]`.
   Дубликаты по `contractorNo` — обновляются, не плодятся.
2. **Свой ВДР в Экселе программы**: таблица-Конструктор помечается «Сделать
   источником ВДР» → её строки читаются в `DocRegisterItem` (маппинг колонок
   сохраняется в `DocRegister.sourceDocId`). Правки таблицы → «Обновить ВДР».
3. **Вручную**: добавить строку реестра в разделе ВДР (для мелких случаев).

### Двусторонние связи (главное требование — «1000 обратных связей»)

**ВДР → документ** (данные втекают в документ):
- при **создании** документа в студии выбираем строку ВДР → автозаполнение:
  номер (contractor/owner/vendor), наименование (Ru/En), категория (vdrCode),
  ревизия, оборудование, реквизиты титульника;
- при **переименовании/присвоении** существующего документа — то же выбором из ВДР;
- «умные поля» в Word (уже есть панель «Данные»): `{ВДР.Номер}`, `{ВДР.Название}`,
  `{ВДР.Ревизия}`, `{ВДР.Тип}` — подставляются и обновляются;
- титульный лист документа генерируется из реквизитов реестра (умный блок «Титул ВДР»).

**Документ → ВДР** (данные втекают в реестр):
- документ можно назначить **источником данных строки ВДР**: «Взять данные из
  документа» → программа читает номер/название/ревизию из шапки документа и
  предлагает обновить строку реестра (с предпросмотром и подтверждением — как
  dry-run импорта бланков);
- смена ревизии/статуса документа → строка ВДР обновляется, дата выпуска ставится;
- новый документ, которого нет в реестре → предложить «Добавить в ВДР».

**Связь по оборудованию**: строка ВДР ↔ теги/оборудование (выборка из ВДР — по
`equipmentTags`). Из карточки оборудования видно «документы по этой позиции».

### Формирование документа (мало действий)
Одна кнопка «Новый документ по ВДР» в студии:
1. выбор строки реестра (поиск по номеру/названию/типу) — или «без ВДР»;
2. авто-подстановка: имя файла = номер, наименование, тип, ревизия, оборудование;
3. выбор шаблона по типу (например, для E02 — шаблон опросного листа);
4. документ создан, привязан к строке ВДР (`fileNodeId`), титул заполнен.
Переименование/ревизия — тоже выбором из ВДР, вручную ничего не печатать.

### Раздел «ВДР» (где живёт)
Отдельный раздел меню **«ВДР» / «Документация»** (или вкладка в Менеджменте):
таблица-реестр (виртуализованная, 562+ строк), фильтры по типу/ревизии/статусу/
исполнителю/оборудованию, счётчики, экспорт в Excel (формат заказчика — как
исходные файлы, с титульником и листом ревизий), массовые действия.

## 2. Процесс и уведомления документооборота

### Роли и маршрут
- **Инженер** формирует документ → ставит статус «Готово».
- **Менеджер** (назначается админом; поле `managerId` у проекта/реестра/строки)
  получает уведомление «Документ X готов» → выгружает/отправляет заказчику
  (сама отправка — вне программы, как сказано).
- **Заказчик вернул замечания** → менеджер грузит файлы в Проводник и **прикрепляет
  к документу** (строке ВДР) → **инженеру-исполнителю** уведомление «Замечания по X».
- Клик по уведомлению → переход к файлу в Проводнике / к строке ВДР.

### Гибкая настройка уведомлений (кому)
Для каждого события документооборота — правило доставки:
- конкретному пользователю (исполнителю/менеджеру),
- роли (все менеджеры / все инженеры дисциплины),
- всем участникам проекта.
Настройка — в «Настройки → Уведомления» (матрица «событие × кому»), плюс
per-проект переопределение (менеджер проекта). События:
`doc.ready` (готов), `doc.remarks` (замечания), `doc.revision` (новая ревизия),
`doc.issued` (выдан заказчику), `doc.assigned` (назначен исполнитель).

### Реализация
- используем существующую таблицу `Notification` (есть) + socket-события (есть);
- добавляем категорию `ДОКУМЕНТЫ`;
- `targetRoute` уже поддерживается (клик → переход) — ведём на `/explorer?file=…`
  или `/vdr?item=…`.

## 3. Уведомления: «Общие» и «Личные»

Сейчас панель справа называется «Уведомления», категории: СИСТЕМА, ОБОРУДОВАНИЕ,
ЧАТ, ПРОЕКТЫ, ДОСТУП.

Новая структура (две вкладки в панели):
- **Общие** — то, что касается всех/проекта: новые релизы, конфликты
  оборудования, изменения в общих данных, публикации ВДР, системные.
- **Личные** — адресованные лично мне: замечания по моему документу, назначение
  исполнителем, ответы в чате, мои сроки доступа.

Разнесение по подразделам (внутри каждой вкладки — группы категорий):
```
Общие:   Система · Обновления · Оборудование · Проекты · ВДР (общее)
Личные:  Мои документы · Назначения · Чат · Доступ
```
Технически: у `Notification` уже есть `userId` (личные) — раскладка по
`scope = personal|shared` + `category`. Заголовок панели → «Общие», рядом
переключатель «Личные». Настройки — какие категории показывать в каждой.

## 4-б. Сценарии использования («1000 сторон») и крайние случаи

**Инженер, обычный день:**
1. Открывает ВДР → фильтр «мои + статус Черновик» → видит, что должен выпустить.
2. «Новый документ по ВДР» на строке E02-0037 → документ создан с номером,
   названием, титулом; чертит опросный лист; статус «Готово» → менеджер уведомлён.
3. Пришли замечания → уведомление «Замечания по E02-0037» → клик → файл замечаний
   в Проводнике рядом с документом → правит → новая ревизия B → снова «Готово».

**Менеджер:**
1. Панель ВДР: счётчики «готово к отправке: 12, у заказчика: 40, замечания: 3».
2. Уведомление «готов» → открывает документ → экспорт PDF/DOCX → отправляет
   (вне программы) → статус «Выдан», ревизия и дата легли в реестр автоматически.
3. Вернулись замечания: тянет файл в Проводник → «Прикрепить к документу…» →
   выбирает строку ВДР → инженеру уведомление.
4. Ежемесячный отчёт: экспорт ВДР в Excel в формате заказчика (титул + ревизии +
   реестр) одним нажатием.

**Админ:** назначает менеджера проекта; настраивает матрицу уведомлений
(например, «замечания» — только исполнителю; «готов» — менеджеру и его заму).

**Ревизии (важная логика из реальных файлов):**
- буквенные A,B,C… = черновые/на рассмотрение (IFR), цифровые 0,1,2… = утверждённые (CEF);
- «повысить ревизию» — одна кнопка: A→B, J→0 (первое утверждение), 3→4;
  причина выпуска подставляется по типу перехода; дата = сегодня;
- история ревизий строки хранится (лист «Accounting for revisions» при экспорте
  собирается из неё).

**Крайние случаи (продуманы):**
- документ без строки ВДР — живёт как раньше; в любой момент «Добавить в ВДР»;
- строка ВДР без документа — «документ ещё не начат», видно в фильтре;
- два инженера на один документ — исполнителей может быть несколько (assignees);
- переимпорт обновлённого Excel-ВДР — сверка по contractorNo: новые добавить,
  изменённые показать diff (как dry-run бланков), исчезнувшие пометить «выпал из
  реестра», ничего не удалять молча;
- «сделать источником» таблицу с кривыми колонками — мастер сопоставления
  колонок с предпросмотром, ничего не пишется без подтверждения;
- документ-источник изменили → строка ВДР помечается «данные разошлись» →
  кнопка «Обновить из документа» с diff и подтверждением (как конфликты умных блоков);
- удаление документа — связь рвётся, строка остаётся (реестр первичен);
- офлайн (встроенный сервер) — всё работает, уведомления локальные;
- 562+ строк — виртуализация таблицы (уже используем @tanstack/react-virtual).
