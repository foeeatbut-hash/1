# Аудит готовности программы к онлайну (центральный сервер компании)

Целевая картина: **один сервер компании** (Express + PostgreSQL + socket.io + файлы),
все сотрудники — клиенты (Electron-приложение или просто браузер: сервер уже раздаёт
собранный фронтенд статикой, `server.ts:3542`). Сегодняшний офлайн (SQLite, встроенный
сервер на localhost) — тестовый стенд, и он остаётся рабочим как частный случай:
«сервер компании» = «мой localhost».

Аудит по коду v0.24: что уже готово к онлайну, что сломается и как чинить.
Итог — в конце, планом из 5 шагов.

---

## 1. Что уже готово (менять не нужно)

| Подсистема | Почему готова |
|---|---|
| Реестр тегов, оборудование, справочники, проекты, закупки, заметки, логи, уведомления, настройки | всё через HTTP API + Prisma; при общей БД и центральном сервере работает как есть |
| Файлы Проводника | содержимое хранится base64 в БД (`FileNode.content`) — едет через общую БД; см. §5 про размеры |
| Пароли | хешируются, легаси-пароли мигрируются при входе (`server.ts:1044-1050`) |
| Автообновления | записи `AppUpdate` в общей БД + скачивание по URL — центральный сервер только упрощает (кладём exe рядом) |
| Тонкий клиент из браузера | сервер раздаёт `dist/` и `index.html` — на офисном сервере программа открывается по `http://server:3000` без установки |
| Экраны с поллингом (чат, уведомления) | опрос по HTTP работает и через сеть; после починки сокетов (§2) станет push, поллинг остаётся страховкой |
| Конструктор (части I–IV) | спроектирован сразу под центральный сервер |

## 2. Сломано: реалтайм-события никогда не ходят между пользователями

`ENV_CONFIG.isLocalMode` (`src/config/env.ts:8`) в Electron **всегда true** (при
`file://` hostname пустой, а «прод» определяется как hostname === 192.168.1.100) →
`SocketProvider` создаёт **LocalMockSocket** — заглушку с локальным эхо. Значит уже
сейчас в «сетевом» варианте (общая PostgreSQL) socket-события (`chat:message_received`,
`equipment:conflict`, обновления списков) не доходят до коллег — экраны живут только
поллингом. Все будущие фичи (совместное редактирование, presence) на этом не построить.

**Чинить (шаг 1 плана):**
- убрать `LocalMockSocket` и «магию» isLocalMode; socket подключается ВСЕГДА к адресу
  сервера (для localhost это соединение с самим собой — работает так же);
- адрес сервера — не захардкоженный `192.168.1.100:5000`, а настройка клиента (§4);
- на сервере события уже шлются (`io.emit`) — доработать только комнаты по проектам,
  чтобы не слать всем всё.

## 3. Сломано: чат в Electron ходит мимо сервера

В собранном приложении `chatStore` предпочитает **IPC-путь**: Electron-main сам лезет
в БД (`electron/main.ts`, обработчики `chat:*`), минуя сервер. Последствия в онлайне:

1. сервер не знает о сообщении → `io.emit('chat:message_received')` не срабатывает →
   собеседник узнаёт о сообщении только поллингом;
2. **вложения пишутся на диск отправителя** (`userData/chat_files` его машины,
   `chat:upload-file`), а «открыть файл» у получателя ищет в его локальной папке —
   файла там нет. Вложения между машинами не работают уже сейчас;
3. дублируется бизнес-логика (уведомления, проверки каналов) в двух местах —
   main-процессе и сервере — они уже разъезжаются.

**Чинить (шаг 2):** удалить IPC-ветку чата целиком — всегда HTTP/WS через сервер.
Серверный путь уже полный: `/api/chat/*` + `/api/chat/upload` кладёт файл на диск
СЕРВЕРА и раздаёт по `/chat_files/...` статикой (`server.ts:678`) — это правильная
модель для онлайна. IPC-обработчики чата в `electron/main.ts` (~400 строк) удаляются,
`getChatPrisma`-прокси тоже. Открытие вложения = скачивание по URL, а не путь на диске.

## 4. Сломано: клиент не умеет «подключиться к серверу компании»

Сейчас у клиента настраивается **база данных** (SQLite-файл или строка PostgreSQL), и
каждый клиент поднимает СВОЙ встроенный сервер поверх общей БД. Для онлайна архитектура
другая: клиент должен знать **адрес сервера**, а какая у сервера БД — не его дело.
Захардкожены: `localhost:3000` в fetch-прокси (`env.ts:41`), `192.168.1.100` как
признак прода и адрес сокета, `updatesUrl`.

**Чинить (шаг 3):**
- в `config.json`/экране входа — поле «Сервер»: `Локально (встроенный)` (по умолчанию,
  = сегодняшнее поведение) или `Адрес сервера компании` (http://host:port);
- fetch-прокси и socket берут базовый URL из этой настройки; выбран удалённый сервер →
  встроенный Express и локальная БД просто не запускаются (мгновенный старт);
- экран «Выбрать файл БД» показывается только для встроенного сервера; настройка БД
  переезжает в админку сервера;
- деплой сервера компании уже возможен без новой сборки: `node dist/server.cjs`
  на офисной машине (Windows-служба/автозапуск) + PostgreSQL. Описать в README.

## 5. Риски (работает, но в онлайне аукнется)

1. **API без авторизации.** Ни один роут не проверяет, КТО спрашивает: любой в офисной
   сети может `curl`-ом читать/менять всё, включая смену паролей. Права (`permissions`)
   проверяются только в интерфейсе. Для LAN-развёртывания это главный риск.
   → Шаг 4: токен сессии при логине (таблица `Session` или подписанный токен),
   `Authorization` в fetch-прокси (одна точка), middleware на сервере: аутентификация
   всех `/api/*` кроме login/health + проверка прав там, где интерфейс их проверяет
   (создание проектов, админка пользователей, удаления). Личные разделы (заметки,
   личные папки, личные документы Конструктора) фильтровать по пользователю ИЗ ТОКЕНА,
   а не из query-параметра, как сейчас.
2. **«Запомнить вход» хранит пароль открытым текстом** в localStorage. → хранить токен.
3. **Файлы Проводника base64 в БД**: лимит запроса 50 МБ ≈ файл до ~35 МБ; большие
   чертежи раздуют PostgreSQL и трафик (каждое скачивание тащит base64 через JSON).
   Приемлемо для старта; следующий шаг — файловое хранилище на диске сервера (как уже
   сделано для вложений чата), в БД — только метаданные. Отметить как «Фаза файлы-2».
4. **Поллинг как основной канал** (чат каждые ~3 с, уведомления): на 30 пользователей —
   сотни запросов в минуту к серверу. После шага 1 поллинг проредить (раз в 30–60 с,
   только как страховка пропущенного события).
5. **Last-write-wins в карточках** (тег, элемент, заметка): двое правят одну карточку —
   побеждает последний, молча. Для карточек это терпимо (правки точечные), но после
   шага 1 добавить дешёвую защиту: событие «карточка изменена» → открытая форма
   показывает «обновлено Ивановым — [обновить]». Полноценный OT — только в Конструкторе.
6. **SQLite-специфика в рантайме сервера**: `busy_timeout`, `connection_limit=1` —
   уже разведено через два Prisma-клиента; проверить, что серверные транзакции не
   полагаются на «однопоточность» SQLite (при PostgreSQL параллельные запросы реальны).
7. **Локальные возможности Electron** (диалоги файлов, печать, стикеры поверх окон,
   crash-логи, захват экрана) — остаются локальными, это правильно; в браузерном
   клиенте их аналоги: стандартная печать, download, без стикеров.

## 6. Сводный план (по убыванию критичности)

| Шаг | Что делает | Статус |
|---|---|---|
| 1. Настоящий socket.io | убран мок и isLocalMode-магия; клиент всегда соединяется с сервером (встроенным или компании) | **✔ внедрено** (v0.24, ветка design) |
| 2. Чат только через сервер | IPC-ветки чата удалены (store, ChatManagement, main.ts); вложения — на сервер, открытие по URL; легаси-открытие старых локальных вложений сохранено | **✔ внедрено** |
| 3. Настройка «адрес сервера» | экран входа: «Встроенный сервер / Сервер компании» с проверкой связи; при удалённом сервере встроенный не запускается; заставка и ServerGate знают про оба случая; README | **✔ внедрено** |
| 4. Авторизация API | подписанный токен при логине (HMAC, 30 дней, без таблиц в БД); middleware на все /api/* (кроме login/health/db-config, /api/db/* — только с машины сервера); socket.io пускает по токену; управление сотрудниками — только ADMIN (сам себе — лишь имя/пароль); 401 → возврат на экран входа | **✔ внедрено** |
| 5. Гигиена реального времени | проредить поллинг; «карточка изменена» в формах; файлы Проводника на диск сервера (фаза 2) | по мере |

Остаётся после шага 4 (осознанно, мелкими шагами):
- **пофункциональные права на сервере** (project.create и т.п. — сейчас
  enforce'ится вход и админ-операции, тонкие права всё ещё проверяет интерфейс);
- **личные данные по токену**: личные заметки/папки пока фильтруются по
  userId из запроса — подменяемо вошедшим пользователем; перевести на
  `req.authUser.id`;
- `/chat_files/*` раздаётся без токена (имена файлов неугадываемые) — при
  необходимости закрыть подписанными ссылками;
- «запомнить вход» ещё хранит пароль в localStorage — заменить на токен.

Как это устроено после шагов 1–3:

- одна настройка на клиенте — `flux_server_url` (localStorage + `remote_server_url`
  в config.json, синхронизируются сами): пусто = встроенный сервер, задано = все
  запросы и socket.io идут на сервер компании, встроенный Express не запускается;
- меняется на экране входа («Встроенный сервер (эта машина)» под формой): адрес,
  «Проверить» (пингует `/api/health`), «Подключиться» → перезагрузка окна;
- сервер недоступен → вместо вечной заставки экран «Сервер компании не отвечает»
  с кнопками «Повторить» и «Встроенный сервер»;
- офлайн-тестирование не изменилось: по умолчанию всё работает как раньше,
  на встроенном сервере этой машины.

Шаг 4 обязателен до развёртывания за пределами доверенного кабинета. Конструктор
(части I–IV) строится поверх шагов 1 и 4 и других предпосылок не имеет.
