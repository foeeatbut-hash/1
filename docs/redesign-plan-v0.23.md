## 1. Блокнот

### 1.1 Ссылки не работают — причина найдена
`RichTextEditor.tsx` вставляет ссылку через `window.prompt()` — **Electron не поддерживает
prompt()**, вызов падает, поэтому кнопка «Вставить ссылку» в приложении мертва.
Клик по готовой ссылке внутри редактора тоже ничего не открывает: contenteditable
перехватывает клик, а в `electron/main.ts` нет обработчика открытия внешних ссылок из окна.

Как будет:
- Вместо prompt — маленькое всплывающее окно у курсора: поле «Адрес», поле «Текст»,
  Enter — вставить. Если выделен текст, он становится текстом ссылки.
- Клик по ссылке в редакторе показывает мини-плашку: адрес + «Открыть» / «Изменить» / «Убрать»
  (как в Word). Ctrl+клик — открыть сразу (через `shell.openExternal`, уже есть IPC
  `shell:open-external`).
- Автоссылки: вставленный из буфера `https://…` сразу становится ссылкой.

### 1.2 Таблица — как в Word
Сейчас: модалка «строк/столбцов» числами, а кнопки «+строку/−столбец» в тулбаре требуют
заранее поставить курсор в ячейку и ругаются `alert()`-ом.

Как будет:
- Кнопка «Таблица» открывает **сетку 10×8**: ведёшь мышью — подсвечивается размер, клик — вставка
  (ровно как в Word). Модалка с числами удаляется.
- Все операции со строками/столбцами переезжают в **контекстное меню ПКМ внутри таблицы**:
  вставить строку выше/ниже, столбец слева/справа, удалить строку/столбец/таблицу,
  заливка ячейки, выравнивание. Кнопки «+строку/−строку/+столбец/−столбец» из тулбара убираются
  (4 кнопки меньше, ноль alert-ов).
- **Tab / Shift+Tab** — переход по ячейкам; Tab в последней ячейке добавляет строку (как в Word).
- Перетаскивание границы столбца мышью — изменение ширины.
- Пустые ячейки вставляются пустыми (сейчас каждая ячейка забита словом «Ячейка», его
  приходится стирать вручную).

### 1.3 Чек-лист — причина глюков
Сейчас вставляется `<input type="checkbox">` внутрь contenteditable. Внутри редактируемой
области Chromium обрабатывает такие инпуты нестабильно: клик то ставит галочку, то ставит
курсор; Enter не продолжает список, а разрывает разметку; состояние теряется.

Как будет — свой блок без `<input>`:
- Пункт чек-листа = строка с нарисованной галочкой (span с `data-checked`), клик по галочке
  переключает и зачёркивает текст. Никаких инпутов внутри contenteditable — глюк исчезает.
- **Enter** в конце пункта создаёт следующий пункт с пустой галочкой; Enter на пустом пункте
  завершает список; Backspace в начале пункта превращает его в обычный текст.
- Прогресс сверху списка: «3/7 выполнено» (считается автоматически).

### 1.4 Мелочи в блокноте
- Цвет текста №1 в палитре — `#0f172a` (почти чёрный): в тёмной теме он невидим.
  Палитра будет двойная: в тёмной теме тёмные цвета автоматически заменяются на читаемые.
- Экспорт в Word: таблицы сейчас уходят без рамок (стили — tailwind-классы, Word их не знает).
  При экспорте классы будут конвертироваться в инлайновые стили (рамки, заливка шапки).

## 2. Менеджмент — лаги (причины найдены)

`ProcurementManagement.tsx`:
1. Таблица рендерит **все** позиции разом — без виртуализации (в «Тегах» виртуализация есть,
   тут нет). На сотнях позиций вход в раздел и прокрутка фризят.
2. `parseMeta()` (JSON.parse metadata) выполняется для каждого тега заново при каждом
   рендере — без кэша `parsedMetadata`, который уже используется в «Тегах».
3. Поиск без debounce: каждый символ → полная фильтрация + сортировка + перерисовка всех строк.
4. Строки таблицы — не мемоизированы: любое изменение одного поля перерисовывает всю таблицу.
5. Массовая установка этапа шлёт запросы **по одному** (`for … await fetch`) — на 50 позициях
   это 50 запросов подряд.

Как будет:
- Виртуализация списка через `@tanstack/react-virtual` (уже в зависимостях).
- Кэш распарсенного metadata на объекте тега (как в Registry).
- Debounce поиска 200 мс + мемоизация строки (`React.memo`).
- Один серверный эндпоинт `PUT /api/tags/bulk` для массовых операций.
- Отказ от анимации входа motion.div на тяжёлом экране.

Ожидаемый результат: вход в раздел и прокрутка — мгновенные при 1000+ позиций.

## 3. Холст (Теги): управление карточками

### 3.1 Упрощение мыши
Текущая модель: ЛКМ по фону — панорама, ЛКМ по карточке — перетаскивание, ПКМ — меню,
Ctrl+клик — выделение. Клик и перетаскивание конфликтуют (карточка «дёргается» при клике).

Новая модель (минимум действий):
| Действие | Результат |
|---|---|
| ЛКМ-клик по карточке | Выделить + мини-панель действий у курсора (см. 3.3) |
| ЛКМ-тащить карточку (порог 5 px) | Перемещение — клик больше не смешивается с drag |
| ЛКМ по фону + тащить | Панорама (как сейчас) |
| ЛКМ по фону + Shift | Рамка-выделение нескольких карточек (lasso) |
| Колесо | Зум к курсору (как сейчас) |
| Двойной клик по карточке | Развернуть/свернуть подробности |
| Ctrl+клик | Добавить/убрать из выделения |
| Esc | Снять выделение / закрыть панели |

### 3.2 «Найти дубль» — по всему разделу Теги
ПКМ по карточке с меткой «дубль» → пункт **«Найти дубли (N)»**:
- **Один дубль** — экран плавно перелетает к нему, карточка 2 сек пульсирует подсветкой.
  Одно действие — ноль поиска глазами.
- **Несколько дублей** — справа выезжает узкая панель «Дубли ABC-123 (4)»: список позиций
  (код, наименование, где находится). **Прокрутка колесом по списку перемещает вид на
  соответствующий дубль** — крутишь ролик, холст сам летает по дублям; наведение на строку
  делает то же. Клик — остаться на нём, Esc — закрыть панель и вернуться.
- Работает одинаково на **холсте** (перелёт камеры), в **дереве связей** (авто-раскрытие ветки,
  скролл к строке, подсветка) и в **спецификации** (скролл к строке, подсветка).
- Та же панель открывается из фильтра «Только дубли» в Менеджменте — кнопкой «Показать на холсте».

### 3.3 Выделение и мини-панель действий
Выделение больше не на ПКМ. Один ЛКМ-клик по карточке = выделение, и справа от курсора
появляется компактная мини-панель (5–6 иконок с подписями при наведении):

- ✚ «Выбрать несколько» — включает режим мультивыбора: дальше каждый клик добавляет
  карточку (без зажатого Ctrl), счётчик выбранного, Esc — выход.
- Связи ↑ (родители) / Связи ↓ (дочерние) — как в текущем ПКМ-меню.
- «Найти дубли» — если у карточки есть дубль.
- «Поделиться в чате», «Редактировать», «Удалить».

Панель исчезает при клике мимо. ПКМ-меню остаётся как дубль тех же действий для привычки.

## 4. Экспорт и импорт — полный редизайн подраздела

Сейчас: два гигантских блока фильтров по сегментам (тег + марка) на весь экран, много
пустоты, заголовков и пояснений; до кнопки экспорта — длинная прокрутка.

Новый макет — всё на одном экране, три компактные зоны:

```
┌────────────┬──────────────────────────────┬───────────────┐
│ ИМПОРТ     │ ФИЛЬТР ПОДБОРКИ              │ ЭКСПОРТ       │
│ · Из Excel │ Сегменты тега:  [3700][C01].. │ ☑ Тег ☑ Марка │
│   (мастер) │ Сегменты марки: [BLM][15]..   │ ☑ Отдел …     │
│ · Из текста│ (чипы; клик — вкл/выкл)       │ [Экспорт .xlsx]│
├────────────┴──────────────────────────────┴───────────────┤
│ ЖИВОЙ ПРЕДПРОСМОТР: таблица результата (виртуализированная)│
│ N позиций · обновляется при каждом клике по чипу           │
└────────────────────────────────────────────────────────────┘
```

- Фильтры по сегментам сворачиваются в строки чипов вместо простыней с подзаголовками;
  привязка к словарям — иконкой на чипе, а не отдельным блоком.
- «Импорт тегов из таблицы» (мастер `TagImportWizard`) остаётся как есть + улучшения:
  запоминание сопоставления колонок последнего импорта, drag&drop файла прямо на карточку.
- Экспорт — сразу в `.xlsx` (библиотека xlsx уже есть), а не CSV: не ломается кириллица в Excel.
- Убираются: длинные подписи-объяснения, дублирующиеся заголовки, градиентные баннеры.

## 6. Рабочий чат — редизайн

Что уже есть и остаётся: личные/групповые чаты, ответы цитатой, редактирование, реакции,
закрепы, пересылка, вложения, скриншот с рисованием, шаринг тегов.

### 6.1 Вид
- Список диалогов: аватар-инициалы с цветом по роли, последнее сообщение, время,
  **счётчик непрочитанных**, точка онлайн-статуса. Компактно, без пустых зон.
- В переписке — разделитель «Новые сообщения» при входе в диалог.
- Правая панель диалога: участники со статусами, закреплённые сообщения, общие файлы,
  теги, упомянутые в диалоге, — вкладками, а не простынёй.

### 6.2 Профили пользователей
Клик по имени/аватару где угодно → карточка профиля: имя, роль, отдел, символ (подпись
на чертежах), онлайн/был в сети, кнопки «Написать» и «Упомянуть». Свой профиль редактируется
(аватар-цвет, отдел, должность, телефон) — данные уже есть в модели User, нужен только UI.

### 6.3 Новые функции — что берём и что нет (продумано по рабочим ситуациям)
Берём:
- **Онлайн-статусы и «печатает…»** (socket.io уже подключён) — видно, кто на месте: важно,
  когда срочно нужен ответ по позиции.
- **Упоминания @имя** с подсветкой и уведомлением — сейчас сообщения легко пропустить.
- **Непрочитанные** (счётчики + последняя прочитанная отметка) — главная боль рабочего чата.
- **Поиск по всем сообщениям** (сейчас поиск только по списку диалогов).
- **«В блокнот»** на любом сообщении — превращает договорённость в заметку одним кликом.
- **Фильтр «Сообщения по тегу»**: из карточки тега видно всё, что о нём писали, — связка
  чата с реестром, которой нет ни в одном мессенджере.
Не берём (осознанно):
- Треды/ветки — команда небольшая, ответ цитатой покрывает задачу, ветки размажут переписку.
- Голосовые сообщения — на производстве их не переслушать, текст ищется, голос нет.
- Сторонние стикеры/гифки — шум.

## 8. Тёмная тема — аудит

Конкретные найденные проблемы:
1. Палитра цвета текста в блокноте: `#0f172a` невидим на тёмном фоне; маркеры-выделения
   (жёлтый/розовый) оставляют светлый фон под светлым текстом. Нужна адаптация цветов к теме.
2. `index.css` определяет нестандартные оттенки (`slate-850`, `slate-450`, `emerald-450`…,
   249 использований) — часть переопределяется в тёмной теме на зеленоватые тона намеренно,
   но из-за этого несколько мест (бордеры `border-slate-855`, текст `text-rose-550`)
   опечатаны и не совпадают с темой. Пройти все нестандартные оттенки, свести к карте темы.
3. Чекбоксы `accent-color` и системные `<select>` в тулбаре редактора не темнеют (нативные
   контролы) — заменить на стилизованные (CustomSelect уже есть).
4. Таблицы, вставленные в заметки в светлой теме, хранят классы `dark:` — ок, но старые
   заметки со «старым» HTML без dark-классов отображаются белым по белому — при открытии
   нормализовать HTML.
5. Скриншот-аннотации в чате: канвас рисует тёмные линии по тёмному скриншоту — выбрать
   контрастный цвет по умолчанию.
6. Системный прогон: скрипт-линт, который находит `bg-*`/`text-*`/`border-*` без парного
   `dark:` в src — и точечная правка найденного (Dashboard, Login, UpdaterWidget — кандидаты).

## 9. Умный импорт из Word и Excel (любые бланки)

Цель: кидаешь любой документ (бланк клапана, вентилятора, приточной установки, ведомость,
опросный лист…) с произвольным расположением данных — программа сама достаёт наименование,
марку, размеры, характеристики и раскладывает в «Оборудование». Полностью локально.

### 9.1 Форматы и извлечение
| Формат | Как читается | Статус |
|---|---|---|
| .xlsx / .xls / .csv | библиотека xlsx (уже есть): все листы → сетка ячеек с учётом объединённых | сразу |
| .docx | mammoth (лёгкая, офлайн): абзацы + таблицы, включая вложенные | сразу |
| .doc (старый) | честное сообщение «пересохраните в .docx» | — |
| .pdf с текстовым слоем | pdfjs-dist (офлайн): текст + координаты → восстановление таблиц | фаза 2 |
| .pdf-скан / фото | текстового слоя нет (проверено на реальных каталогах в проекте — там картинки). Честное сообщение + опционально OCR (tesseract.js, офлайн) отдельной фазой | фаза 3, по решению |

### 9.2 Первый проход — определение типа документа
Стратегия разбора зависит от того, ЧТО это за документ. Классификатор смотрит на форму
таблиц и повторяемость, и относит документ к одному из типов:

1. **Карточка изделия** — один предмет, параметры списком/таблицей (бланк клапана).
2. **Ведомость / спецификация** — таблица, где строки = позиции, шапка = поля.
   Каждая строка становится отдельной единицей оборудования.
3. **Опросный лист** — поля с пропусками, подчёркиваниями, вариантами «нужное отметить».
   Берутся только заполненные значения; пустые поля не тащатся как мусор.
4. **Многосекционный бланк** — несколько изделий в одном документе (приточная установка:
   секции «Вентилятор», «Калорифер», «Фильтр»…). Разбирается по секциям, см. 9.6.
5. **Не распознаваемый** (скан, каталог без структуры) — сразу честный ответ с причиной,
   а не пустой результат.

Признаки: сущностная таблица — в шапке слова из словаря полей, ниже однотипные строки;
атрибутная — словарные слова идут в первой КОЛОНКЕ; опросный лист — подчёркивания,
чекбоксы, «да/нет»; многосекционный — словарь типов оборудования встречается несколько
раз как заголовки.

### 9.3 Анализ таблиц по форме
В Word бланк часто целиком нарисован одной большой таблицей с вложенными — это оформление,
а не данные. Разбор рекурсивный, и каждая таблица классифицируется:

- **Атрибутная** (параметр | значение, вертикальная) → пары «ключ→значение».
- **Сущностная** (шапка полей, строки-позиции) → по позиции на строку.
- **Матричная** (типоразмеры в колонках, параметры в строках — типично для каталогов) →
  программа ищет рядом марку/типоразмер, чтобы выбрать нужную колонку; если не нашла —
  на предпросмотре спрашивает: «какая колонка ваша?» (один клик).
- **Оформительская** (рамка бланка, шапка организации) → пропускается, разбор уходит внутрь.

### 9.4 Зонирование: текст с абзацами + параметры
Документ читается блоками (абзац / таблица / строка листа), каждый блок классифицируется:
- *проза* — длинные предложения, знаки препинания, мало цифр, нет «ключ: значение»;
- *строка-параметр* — короткая, с двоеточием/тире/табуляцией, справа число с единицей
  или код («Расход воздуха: 5000 м³/ч»);
- *шапка/подпись/мусор* («Утверждаю», даты, реквизиты, номера страниц) — отсекается.

Проза не выбрасывается, а дочитывается отдельным проходом: в предложениях часто лежат
наименование, марка, тип оборудования и система («…подбор осевого вентилятора
ОСА-ЭВО-5,6 для системы П1…»). Работают регэксп кодов (уже есть в Registry), словарь
типов оборудования и шаблоны обозначений систем. Прозаические формулировки параметров
(«мощность двигателя составляет 3 кВт») ловятся парой «слово-из-словаря … число+единица»
в пределах предложения. Найденное в прозе получает пониженную уверенность.

### 9.5 Поиск значения по якорю
Подпись поля («Расход воздуха, м³/ч») — якорь. Значение ищется в порядке приоритета:
справа в той же строке → под якорем → справа-снизу (для объединённых ячеек). Единица
измерения подхватывается из самой подписи, из соседней ячейки или из значения. Подписи,
разорванные переносом на две строки/ячейки, склеиваются перед сопоставлением со словарём.

### 9.6 Валидация значений — защита от ложных срабатываний
У каждого поля словаря есть ожидаемый тип значения:
- числовые поля — число + допустимые единицы + разумный диапазон (расход 10…500000 м³/ч);
- марка/код — паттерн «латиница+цифры с разделителями»;
- перечисления (среда, материал) — список допустимых значений из Справочника.

Если под якорь «Расход» попала строка «ВР-80» — проверка типа не проходит, поиск значения
продолжается дальше, а не тащит мусор. Нормализация: десятичная запятая → точка,
«тыс. м³/ч» → ×1000, «400х400» → габарит, похожие кириллические буквы в кодах (С,В,Р,О…)
→ латиница с подсветкой замены (та же защита, что уже стоит на вводе тегов).

### 9.7 Несколько изделий в одном документе
Границы позиций определяются по: заголовкам-секциям из словаря типов («Вентилятор»,
«Калорифер»), появлению нового кода/марки, границам листов Excel, разделительным строкам.
Каждая секция → своя карточка оборудования. Для составных изделий (установка → её секции)
программа сразу предлагает создать **иерархию со связями** — на холсте появляется
родительская карточка установки и дочерние компоненты, уже соединённые линиями.

### 9.8 Связка с существующей базой
- Если распознанный тег/марка **уже есть в проекте** — вместо создания дубля предлагается
  «Обновить характеристики» с наглядным сравнением старое/новое (механизм ревизий
  оборудования уже существует — переиспользуется).
- Если код не найден, но марка совпадает с существующей позицией — предупреждение
  о возможном дубле прямо на предпросмотре.
- При пакетном импорте дубликаты ловятся и **внутри самой пачки**.

### 9.9 Лестница отказоустойчивости — тупика не бывает
1. Уверенный структурный разбор (тип документа определён, поля валидны) → почти всё зелёное.
2. Тип не определён → общий сбор пар «ключ→значение» по всему документу.
3. Пары не нашлись → показываем сырой текст/таблицы, пользователь перетаскивает значения
   в поля карточки вручную (всё равно быстрее ручного ввода — данные уже извлечены).
4. Файл нечитаем (скан) → понятное сообщение, что именно не так и что сделать.

### 9.10 Экран предпросмотра (обязателен всегда)
Данные инженерные — слепой импорт опасен. Слева документ с подсвеченными зонами
(серое — описание, синее — данные), справа — карточки оборудования. Зелёное — уверенно,
жёлтое — проверить, серое — не распознано. Ошиблась граница зоны — клик по абзацу
«это тоже данные», и он перечитывается. Кнопка одна: «Импортировать N позиций».

### 9.11 Память шаблонов
Подтверждённое сопоставление сохраняется по «отпечатку» бланка: нормализованный набор
подписей + форма таблиц + поставщик из шапки. Второй такой же бланк — импорт в один клик
без вопросов. Шаблоны видны и редактируются в Справочнике; исправления пользователя
на предпросмотре дообучают шаблон.

### 9.12 Пакетный импорт
Можно бросить сразу несколько файлов/папку: очередь с общей сводкой — «5 файлов,
12 позиций, 9 готово, 3 требуют проверки» — и переход по жёлтым позициям стрелками.

Это надстройка над существующим импортом XLSX/XML оборудования — старый механизм не ломается.

## 10. ИИ-помощник (локальный ассистент)

Сейчас: поиск по ключевым словам в `knowledge.ts` + простые запросы к данным; падает на
опечатках, склонениях и вопросах «не теми словами».

Как будет:
1. **Нормализация русского**: приведение слов к основе (лёгкий стеммер, без внешних библиотек),
   допуск 1–2 опечаток (расстояние Левенштейна) — «спицификация», «менеджмнт» начинают находиться.
2. **Приоритет живых данных**: если в вопросе распознан код тега/марка/система — сначала ответ
   из базы (карточка, статус, дубли, этап закупки, «показать на холсте»), потом справка.
3. **Действия-кнопки в ответе**: «Открыть на холсте», «Выгрузить в Excel», «Показать в
   Менеджменте» — вместо текста «зайдите в раздел…».
4. **Навигационные команды**: «открой менеджмент», «создай заметку», «найди дубли».
5. **Контекст диалога**: помнит последний предмет разговора («а сколько их?» после вопроса
   о тегах вентиляции).
6. **Честное «не знаю»** с 3 подсказками тем вместо нерелевантного ответа с низким скором
   (сейчас порог 1.5 пропускает мусорные совпадения).
7. Чистка базы знаний: убрать устаревшие ответы, привести названия разделов к актуальным,
   добавить статьи по новым функциям из этого плана.
