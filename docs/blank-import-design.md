# Импорт бланков 2.0: считывание данных и формирование их в программе

Проектирование (без внедрения). Цель — чтобы **любой** бланк расчёта/опросный лист
попадал в программу с минимумом ручной работы и максимумом доверия: инженер видит,
ЧТО именно распознано, ЧТО изменится в проекте, и подтверждает — а не «загрузил и
молится». Документ опирается на разбор текущего кода: `server/equipmentParser.ts`,
`server/equipmentImport.ts`, `server/excelParser.ts`, `src/import/*` (извлекатели
PDF/Word/Excel + офлайн-OCR), `DocImportWizard`, роуты `/api/equipment/import-*`.

---

## 1. Что есть сейчас — честный диагноз

Пайплайн: файл (из Проводника или мастера) → парсер → сразу запись в БД
(`importEquipmentToDB`) с диффом по параметрам и конфликтами immediate/wait.
Клиентский мастер (`DocImportWizard`) умеет PDF/Word/Excel/XML/буфер и офлайн-OCR
сканов (tesseract, пул воркеров) — это сильная база. Но:

### 1.1. Считывание (парсер) — жёсткие эвристики одного формата

| Место | Как сейчас | Где ломается |
|---|---|---|
| Оглавление книги | лист «0»/«index»/первый, коды по regex `у\d`, `мн\d`, `бл\d` | бланки поставщиков (Веза, НЕД, Systemair, VentCalc, MagiCAD) с другой структурой уходят в фоллбэк «каждый лист — блок» и теряют иерархию |
| Поиск листа блока | точное имя или `бл{номер}` | «Бл 2.1 (клапан)», лист с пробелом/скобками — не найден, блок без параметров |
| Строка группы | ровно 1 непустая ячейка | объединённая ячейка заголовка с «хвостом» в соседней колонке; группа с примечанием во 2-й колонке → превращается в параметр |
| Параметр | фиксированно колонки 0/1/2 = ключ/значение/ед | бланки с колонками «№, Параметр, Ед., Значение», с двумя колонками значений (приток/вытяжка), с вертикальной ориентацией — читаются мимо |
| Значения ячеек | `sheet_to_json` без `raw:false` | **даты приходят серийными числами Excel** (45213 вместо 12.10.2023); форматированные числа теряют вид («0,50» → 0.5); формулы — только вычисленное значение без предупреждения |
| Объединённые ячейки | не разворачиваются | шапки на 2 строки и «лесенки» дают пустые ключи |
| Тип оборудования | ключевые слова в названии | «панель» → ВОЗДУХОПРИЁМНЫЙ (ложные срабатывания на «панель управления») |
| XML | разбор регулярками | CDATA, namespace, вложенные элементы, кавычки-апострофы — молча теряются |
| Шапка «Параметр/Значение/ед» | точный словарь слов | «Наименование показателя», «Величина», «Ед. изм.:» — не распознаются, попадают в параметры |

### 1.2. Формирование (запись в проект)

- **Нет предпросмотра для серверного пути**: `/api/equipment/import-to-category`
  пишет в БД сразу; инженер узнаёт о кривом распознавании постфактум по конфликтам.
- **Сопоставление по имени**: `EquipmentSystem` ищется по `name+category` —
  переименованная в новой ревизии установка создаёт дубль; «у1» и «У1» — два разных.
- **Нет отчёта импорта**: сводка (`systems/new/updated/conflicts`) есть, но не
  сохраняется — через неделю не восстановить, что принёс конкретный файл.
- **Нет отката**: импорт с неверным профилем перемешивает параметры — вычищать руками
  (история `EquipmentHistory` есть, но восстановление поштучное).
- **Ключи параметров не нормализуются**: «Расход воздуха», «Расход, м3/ч»,
  «Производительность» из разных бланков = три разных параметра. Мы лечим это
  алиасами в Конструкторе — но лечить надо у входа, а не после.
- **Единицы не канонизируются**: м3/ч, м³/ч, m3/h — разные строки; Па/кПа не
  приводятся; «мм.вод.ст.» живёт как есть.
- **Теги не связываются**: если в бланке есть колонка/поле «Тег» (KKS), связь
  элемент↔тег всё равно делается руками.

## 2. Целевая архитектура: конвейер с профилями и предпросмотром

```
Источник (xlsx/xml/pdf/скан/буфер)
  → ИЗВЛЕЧЕНИЕ         единая промежуточная модель (сетка листов + мета ячеек)
  → РАСПОЗНАВАНИЕ      профиль бланка: где иерархия, где параметры (§4)
  → НОРМАЛИЗАЦИЯ       числа, даты, единицы, синонимы ключей (§5)
  → СОПОСТАВЛЕНИЕ      с существующими сущностями проекта, dry-run дифф (§6)
  → ПРЕДПРОСМОТР       дерево + дифф + предупреждения; правки инлайн (§7)
  → ИМПОРТ             транзакция + снимок «до» + отчёт с переходами (§6.4)
```

Ключевой принцип: **парсер никогда не пишет в БД**. Он выдаёт разобранное дерево с
оценками уверенности; запись — отдельный шаг после подтверждения человеком (для
доверенных профилей — с автоподтверждением, §7.3). Оба существующих пути (файл из
Проводника и клиентский мастер) сводятся к одному конвейеру.

### 2.1. Промежуточная модель извлечения

Сегодня парсер работает по «голым» строкам. Нужна модель с метаданными ячеек:

```ts
interface XCell { v: any; w: string; f?: string; merged?: {rows,cols}; style?: {bold} }
// v — сырое значение, w — как ОТОБРАЖАЛОСЬ в Excel (даты/форматы!), f — формула
```

- читаем `sheet_to_json` с `raw:false` **и** `raw:true` (w и v), храним оба:
  сопоставление и показ — по `w`, вычисления — по `v`;
- **merged-ячейки разворачиваются** (значение копируется во весь диапазон) — ключи
  и заголовки перестают «пустеть»;
- `style.bold` из xlsx — сильный сигнал «это заголовок группы» (надёжнее, чем
  «одна непустая ячейка»);
- для PDF/сканов ту же модель выдаёт существующий извлекатель `src/import`
  (таблицы из OCR уже собираются) — конвейер один для всех форматов.

## 3. Улучшение считывания: колонки и структура по содержимому, не по позиции

### 3.1. Автоопределение колонок таблицы параметров

Вместо «0/1/2 = ключ/значение/ед» — классификатор колонок по содержимому:

- колонка, где >70% ячеек — текст с буквами и без чисел → **ключ**;
- колонка, где >60% — числа/числа с хвостом → **значение**;
- колонка коротких строк из словаря единиц (м, мм, м³/ч, Па, кВт, °C, %, об/мин,
  кг, дБ(А)…) → **единица**;
- колонка «№», порядковые числа → игнор;
- **две числовые колонки** (приток/вытяжка, зима/лето) → два набора параметров с
  суффиксом из заголовка колонки: «Расход воздуха (приток)» / «…(вытяжка)»;
- шапка таблицы находится по строке, где эти классы «переключаются», а не по
  словарю точных слов (словарь остаётся как подсказка).

### 3.2. Структура листа

- заголовок группы: жирная строка / merged на всю ширину / строка-«не-параметр»
  между таблицами — комбинация сигналов со счётом, а не одно правило;
- **вертикальные бланки** (параметры в строке-шапке, значения под ней — частый вид
  опросных листов): детект «шапка из текстов + ряды значений» → транспонирование;
- многостраничные листы: повтор шапки таблицы внутри листа не создаёт новую группу;
- «шум» (логотипы, подписи, рамки штампа) — строки с одиночными короткими текстами
  по краям отбрасываются по эвристике плотности.

### 3.3. Форматы

- **XML — настоящий парсер** (fast-xml-parser, MIT, 50 КБ) вместо регулярок:
  CDATA/namespace/вложенность перестают терять данные; текущие теги-синонимы
  (system/установка/unit…) остаются как маппинг;
- **PDF-бланки и сканы**: конвейер `src/import` (pdfjs + tesseract) подключается к
  импорту оборудования — сегодня он работает только в DocImportWizard. Скан
  расчёта → OCR-таблицы → та же промежуточная модель → тот же профиль;
- **буфер обмена**: вставил таблицу из письма поставщика → тот же путь.

## 4. Профили бланков — сердце улучшения

Разные поставщики = разные структуры. Вместо одного «зашитого» формата — библиотека
**декларативных профилей**:

```jsonc
{
  "id": "vezacalc-v2",
  "name": "Расчёт Веза (VezaCalc)",
  "detect": {                    // как узнать формат (счёт совпадений)
    "sheetNames": ["0", "у*"],
    "cellSignatures": [{ "sheet": 0, "cell": "A1", "contains": "ВЕЗА" }]
  },
  "hierarchy": {                 // где иерархия у/мн/бл
    "source": "indexSheet",      // indexSheet | sheetNames | inSheet
    "unitRe": "^[уy]\\d+", "monoRe": "^мн\\d+", "blockRe": "^бл\\d+"
  },
  "paramTable": { "columns": "auto" },   // или явные индексы колонок
  "tagField": { "group": "Общие", "key": "Тег|KKS|Обозначение" },  // автосвязь тегов
  "typeRules": [ ... ]           // переопределения detectEquipType
}
```

1. **Автоопределение**: при загрузке файла все профили пробуют `detect`, побеждает
   лучший счёт; текущая логика (`у/мн/бл` + оглавление) становится профилем
   `generic-vent-calc`, фоллбэк «каждый лист — блок» — профилем `flat-sheets`.
   Инженер видит «Распознан формат: Расчёт Веза» и может переключить вручную.
2. **Поставляемая библиотека**: стартовые профили под реальные бланки, которыми
   пользуется отдел (собрать корпус образцов — §9). Профили — JSON в БД
   (`AppSetting` или таблица `ImportProfile`), правятся без пересборки программы.
3. **Обучение «по образцу»** — киллер-фича: формат не распознался → мастер открывает
   лист и просит разметить мышью один пример («это код блока», «это таблица
   параметров: ключ/значение/ед», «это название установки») → из разметки
   генерируется профиль → применяется ко всему файлу → инженер проверяет →
   профиль сохраняется под именем и дальше срабатывает сам. Один раз разметил —
   навсегда научил.
4. Профили **общие для отдела** (живут в БД сервера компании) — научил один,
   работает у всех.

## 5. Нормализация — один словарь на всю программу

Выполняется после распознавания, до сопоставления. Всё — данные, не код:

1. **Числа**: `parseRuNumber` уже есть (Конструктор) — выносится в общий модуль;
   «1 250,50», «1.250,5», диапазоны «100…120» (хранится строкой + мин/макс).
2. **Даты**: серийные числа Excel → ISO при извлечении (сегодня теряются!).
3. **Единицы — канонический словарь**: `м3/ч|м³/ч|m3/h|м3/час → м³/ч`;
   `Па|kPa→Па (×1000)|мм.вод.ст.→Па (×9.81)` — с пересчётом значения ТОЛЬКО как
   подсказкой в предпросмотре («привести 1,2 кПа → 1200 Па?»), автоматически не
   пересчитываем (инженерное недоверие к тихим конверсиям оправдано).
4. **Синонимы ключей параметров** — та же таблица алиасов, что у Конструктора
   (`constructor_param_aliases`), применяется **на входе**: «Производительность»
   из бланка Везы сразу ложится ключом «Расход воздуха», и колонки в Конструкторе
   перестают быть дырявыми. Двусторонняя выгода: алиасы, созданные в Конструкторе,
   улучшают импорт, и наоборот — мастер импорта предлагает создать алиас, увидев
   «похожий» ключ (Левенштейн по нормализованной основе).
5. **Валидация значений** — правила на тип оборудования (данные, не код):
   `ВЕНТИЛЯТОР.Расход: number 10..500000 м³/ч`, `Высота: 10..5000 мм`. Нарушение —
   не ошибка, а **жёлтая пометка в предпросмотре** («Высота = 12 500 мм — проверьте»).
   Ловит съехавшие колонки и ошибки OCR лучше любых эвристик.
6. **Мусор OCR**: `textQuality` из `src/import/dictionary` уже оценивает качество —
   порог + подсветка сомнительных ячеек в предпросмотре (уверенность OCR по словам
   уже возвращается из пула воркеров).

## 6. Формирование в программе: сопоставление, dry-run, отчёт, откат

### 6.1. Сопоставление с существующим (вместо «по имени или дубль»)

- нормализация кода: `у1 == У1 == y1 == У-1` (регистр/латиница-кириллица/дефисы);
- **fuzzy-кандидаты**: установка не нашлась по коду → кандидаты по названию
  (нормализованный Левенштейн ≥ 0.75) → в предпросмотре выбор: «это У1
  (переименована)?» или «новая установка»;
- сопоставление блоков внутри моноблока — по `itemCode`, затем по названию+типу;
- решения сопоставления запоминаются в профиле файла (`fileName`) — повторная
  ревизия того же расчёта проходит без вопросов.

### 6.2. Dry-run — обязательный шаг

Серверный `importEquipmentToDB` разделяется на «рассчитать план» и «применить план»:

```
план = { системы: [{ действие: create|match, ... }],
         блоки: [{ действие: create|update|unchanged, изменений: N }],
         конфликты: [...], пропало из бланка: [...],
         связи тегов: [{ identifier, элемент, действие: link|create-tag }],
         предупреждения: [валидация §5.5, OCR-уверенность, непризнанные листы] }
```

План показывается в предпросмотре (§7) и применяется только после подтверждения —
одной транзакцией.

### 6.3. Ручные overrides — уважать

Сегодня импорт кладёт новые specs, а ручные правки инженера живут в `overrides` и
молча «прячут» обновлённые значения. В плане импорта — отдельная секция «затронет
ваши ручные правки: N» с разрешением как в Конструкторе (моё/из бланка, поштучно
и массово).

### 6.4. Отчёт и откат

- **Отчёт импорта** сохраняется (таблица `ImportRun`: файл, профиль, автор, план,
  итог) и виден в истории раздела: «что принёс этот файл» — навсегда, с
  кликабельными переходами к элементам (`targetRoute`, как в `SystemChangeLog`);
- **Снимок до импорта**: перед применением — версия specs затронутых элементов
  (механика `EquipmentHistory` уже пишет oldSpecs/newSpecs) + кнопка **«Откатить
  импорт целиком»** на отчёте (обратное применение плана, доступно, пока не было
  следующего импорта той же области);
- socket-событие после импорта уже есть (fingerprint Конструктора среагирует —
  открытые документы покажут «данные изменились»).

### 6.5. Автосвязывание тегов

Если профиль знает поле тега (§4 `tagField`) или в параметрах распознан ключ
«Тег/KKS/Обозначение»: план содержит связи `элемент ↔ Tag.identifier`;
несуществующие теги — опция «создать недостающие теги» (с шаблоном генератора).
Закрывает самый нудный ручной шаг после каждого импорта.

## 7. UX: один мастер на все источники

`DocImportWizard` становится единственной точкой входа (кнопка «Импорт» в
Оборудовании и перетаскивание файла на раздел):

1. **Источник**: drag&drop / Проводник программы / буфер. Сразу после чтения —
   бейдж «Формат: Расчёт Веза (уверенность 92%)» с ручным переключателем профиля.
2. **Предпросмотр-дерево**: слева у→мн→бл со счётчиками параметров и значками
   (новый/изменится/конфликт/предупреждение), справа — параметры выбранного блока
   *как распознаны*, с подсветкой: жёлтое — валидация/OCR-сомнение, синее —
   изменит существующее, красное — конфликт с ручной правкой. Инлайн-правка любой
   ячейки прямо здесь (исправить кривой OCR до импорта, а не в карточке после).
3. **Область импорта**: галочки на узлах дерева — импортировать только выбранное
   (частая нужда: из большого расчёта нужна одна установка).
4. **Кнопка «Импортировать N блоков»** → транзакция → отчёт с переходами.
5. **Не распознался**: вместо кривого фоллбэка — «Разметить образец» (§4.3).
6. **Доверенный профиль**: у профиля флаг «импортировать без предпросмотра, если
   нет конфликтов и предупреждений» — для ежедневных однотипных ревизий путь
   сокращается до «перетащил → тост с отчётом».

Правило одного диалога (как в Конструкторе): весь мастер — один экран с шагами,
никаких вложенных подтверждений.

## 8. Модель данных (дельта)

```prisma
model ImportProfile {  // профили бланков (§4)
  id String @id @default(uuid())
  name String
  kind String @default("EQUIPMENT")  // EQUIPMENT | TAGS | DOCS
  definition String                   // JSON профиля
  trusted Boolean @default(false)     // импорт без предпросмотра
  createdById String?
  updatedAt DateTime @updatedAt
}

model ImportRun {      // отчёты импортов (§6.4)
  id String @id @default(uuid())
  projectId String
  fileName String
  profileId String?
  authorId String?
  plan String          // JSON плана (что распознано/что применено)
  summary String       // JSON сводки (счётчики)
  rolledBack Boolean @default(false)
  createdAt DateTime @default(now())
}
```

Плюс общий модуль нормализации `server/normalize.ts` (числа/даты/единицы/основы
слов) — используется импортом, Конструктором и ИИ-ассистентом (стеммер из его
дизайна — тот же корень).

## 9. Тесты: корпус бланков — главный актив

1. **Корпус реальных файлов** (обезличенных): по 1–2 образца каждого формата,
   которым пользуется отдел, в `tests/fixtures/blanks/`. Каждому — эталонный JSON
   разбора. Прогон на каждом изменении парсера: «золотые файлы» ловят регрессии
   лучше любых юнитов.
2. **Юниты**: классификатор колонок (§3.1) на таблице образцов; нормализация единиц;
   транспонирование; merged-ячейки; серийные даты.
3. **Ломаные файлы**: пустая книга, книга без параметров, лист на 100 000 строк
   (лимит + честная ошибка), xlsx с паролем, битый zip — парсер отвечает
   диагностикой, не 500-кой.
4. **OCR-план**: 2–3 скана разного качества с эталоном; порог уверенности, при
   котором ячейка обязана подсветиться.

## 10. Фазы внедрения

| Фаза | Содержание | Эффект |
|---|---|---|
| 1. Гигиена парсера | `raw:false` + серийные даты, разворот merged, настоящий XML-парсер, автоколонки §3.1, жирность как сигнал группы | большинство «почти наших» бланков начинают читаться правильно — без новых понятий |
| 2. Dry-run + предпросмотр | план импорта, дерево с диффом и инлайн-правкой, область импорта, уважение overrides | конец «загрузил и молись»; кривой импорт умирает до БД |
| 3. Профили | детект, библиотека, generic-профили из текущих эвристик, ручной выбор | чужие форматы перестают быть болью |
| 4. Нормализация | единый словарь единиц/синонимов с Конструктором, валидация значений | параметры сшиваются между бланками; Конструктор получает целые колонки |
| 5. Отчёты и откат | ImportRun, снимок до, «откатить импорт», автосвязь тегов | доверие + аудит |
| 6. Разметка по образцу + доверенные профили | обучение мышью, автоимпорт без предпросмотра | «перетащил — готово» для ежедневных ревизий |
| 7. Сканы/PDF в оборудование | конвейер src/import подключается к профилям | бумажные опросные листы поставщиков |

Фазы независимы и полезны по отдельности; 1–2 — быстрые и снимают основную боль.
